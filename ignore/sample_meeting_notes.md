## 1. 회의 배경 및 안건 소개

이지훈(PM): 오늘 회의를 소집한 이유는 그린커머스 측에서 우리가 납품한 CS-Bot에 대해 공식적으로 유지보수를 요청해왔기 때문입니다. 지난주 금요일에 그린커머스 IT 담당자 김민수 팀장으로부터 메일을 받았고, 어제 전화 통화로 구체적인 불만 내용도 들었습니다. 오늘 회의에서는 고객의 불만 내용을 공유하고, 우리가 우선적으로 확인해야 할 것들을 정리하는 것이 목표입니다.

고객사 불만 요약은 다음과 같습니다. 첫째, 챗봇이 고객 질문에 엉뚱한 답변을 자주 한다고 합니다. 특히 반품 및 환불 관련 질문에서 오답률이 높다는 구체적인 지적이 있었습니다. 둘째, 답변이 너무 길고 형식적이어서 실제 고객 응대에 어울리지 않는다는 피드백도 있었습니다. 셋째, 간혹 질문과 전혀 무관한 내용을 출력하는 할루시네이션이 발생한다고 합니다. 고객사는 이 문제가 지속되면 계약 연장 여부를 재검토하겠다는 의사를 내비쳤습니다.

## 2. 현황 파악 및 토론

**박서연(AI 엔지니어):** 납품 시점은 작년 11월이었습니다. 당시 사용한 모델은 GPT-4o 기반이었고, RAG 파이프라인을 구성해서 그린커머스의 FAQ 문서와 내부 정책 문서를 지식 베이스로 활용했습니다. 지식 베이스 문서는 납품 당시 기준으로 최신화된 것이었는데, 현재 그린커머스의 반품·환불 정책이 변경되었을 가능성이 있습니다.

**최민준(백엔드):** 납품 이후 저희 쪽에서 별도로 업데이트를 진행한 적은 없습니다. 서버는 잘 돌아가고 있었고, 로그는 쌓이고 있는 상태인데 따로 분석을 한 적은 없었습니다.

**윤재혁(QA):** 로그를 먼저 봐야 할 것 같습니다. 어떤 질문 유형에서 오답이 많이 발생하는지 패턴을 확인하는 게 가장 먼저입니다. 고객사가 반품 관련 질문을 특정해서 말씀해 주셨는데, 실제 로그에서도 그게 맞는지 확인해야 합니다.

**강다은(프론트엔드):** 저는 UI 쪽에서도 이슈가 있을 수 있다고 생각합니다. 고객이 질문을 어떻게 입력하고 있는지, 맥락이 제대로 전달되고 있는지 프론트 로그도 같이 봐야 할 것 같습니다.

**이지훈(PM):** 그러면 지금 당장 확인해야 할 항목들을 정리해 봅시다. 크게 세 가지 레이어로 나눠볼 수 있을 것 같습니다. 첫 번째는 데이터 레이어인데, 지식 베이스 문서가 최신 상태인지 확인하는 것입니다. 두 번째는 모델·파이프라인 레이어로 RAG 검색 품질과 프롬프트 설계를 점검하는 것입니다. 세 번째는 시스템 레이어로 응답 로그 분석과 에러 패턴 파악입니다.

**박서연(AI 엔지니어):** 추가로 모델 버전도 확인해야 합니다. 납품 당시 GPT-4o였는데, 그 이후로 OpenAI에서 여러 업데이트가 있었을 수 있고, API 동작 방식이 바뀌었을 수도 있습니다.

**윤재혁(QA):** 고객사에서 직접 수집한 불만 사례가 있다면 그것도 요청해야 할 것 같습니다. 고객들이 실제로 어떤 질문을 했고 어떤 답변이 나왔는지 구체적인 케이스를 받아서 분석하면 훨씬 빠를 것 같습니다.

## 1. 지난 회의 액션 아이템 이행 확인

이지훈(PM): 지난 회의 이후 일주일간 각자 진단 작업을 진행했습니다. 오늘은 결과를 공유하고 원인을 분석한 뒤 개선 방향을 논의하겠습니다.

고객사로부터는 최신 정책 문서를 수령했습니다. 다만 로그 데이터 공유는 개인정보 처리 문제로 익명화 작업이 필요하다고 해서, 이번 주 중으로 받기로 했습니다. 계약 범위 확인 결과, 현재 요청된 유지보수는 하자보수 기간 내에 해당하므로 추가 비용 없이 진행 가능합니다.

## 2. 각 담당자 진단 결과 보고

**최민준(백엔드) — 서버 로그 분석 결과:**  
6개월치 로그를 분석했습니다. 하루 평균 약 340건의 질의가 들어오고 있고, 이 중 응답 시간이 5초를 넘는 케이스가 약 12%였습니다. 더 중요한 건 "답변을 찾을 수 없습니다"라는 fallback 응답 비율이 전체의 18%에 달한다는 점입니다. 반품 및 교환 관련 키워드가 포함된 질문의 경우 fallback 비율이 31%까지 올라갔습니다. 이건 RAG 검색 단계에서 관련 문서를 제대로 못 찾고 있다는 신호로 보입니다.

**박서연(AI 엔지니어) — 파이프라인 및 모델 진단 결과:**  
현재 파이프라인은 납품 당시 그대로입니다. 문제를 크게 세 가지로 정리했습니다. 첫째, 지식 베이스 문서가 작년 10월 기준이어서, 그린커머스가 올해 2월에 변경한 반품 정책이 전혀 반영되어 있지 않습니다. 반품 가능 기간이 7일에서 14일로 바뀌었는데 챗봇은 여전히 7일이라고 안내하고 있었습니다. 둘째, chunk 분할 방식이 고정 길이 기반이라 문서의 맥락이 끊기는 경우가 많았습니다. 특히 정책 문서처럼 조건이 복잡한 내용은 chunk가 잘리면서 앞뒤 조건이 분리되는 문제가 있었습니다. 셋째, 프롬프트가 납품 시점 이후 한 번도 수정되지 않아 현재 응답 길이나 톤이 고객사 요구에 맞지 않습니다.

**윤재혁(QA) — 자체 테스트 결과:**  
반품, 환불, 교환 관련 질문 20개를 직접 테스트했습니다. 이 중 9개에서 오답 또는 불완전한 답변이 나왔습니다. 오답률이 45%입니다. 틀린 답변의 대부분은 구 정책 기준으로 답하는 경우였고, 일부는 전혀 관계없는 배송 안내를 반품 질문에 답하는 케이스도 있었습니다. 후자는 명백한 검색 오류입니다.

**강다은(프론트엔드) — 사용자 입력 패턴 분석 결과:**  
프론트 로그를 보면 고객들이 질문을 매우 짧고 구어체로 입력하는 경우가 많습니다. "반품 어떻게 해요", "환불 되나요" 같은 식입니다. 현재 RAG 파이프라인은 이런 짧은 질문에 대한 검색 최적화가 되어 있지 않은 것으로 보입니다. 또한 채팅 창에서 이전 대화 내용이 맥락으로 넘어가지 않아서 같은 내용을 반복해서 물어보는 패턴이 보입니다.

## 3. 원인 정리 및 우선순위 논의

이지훈(PM): 지금까지 나온 원인을 정리하면 크게 세 가지입니다.

- **원인 A (가장 시급):** 지식 베이스 문서 미갱신 — 정책이 바뀌었는데 문서가 안 바뀜
- **원인 B (중요):** RAG 검색 품질 — chunk 분할 방식과 검색 알고리즘 개선 필요
- **원인 C (중요):** 프롬프트 개선 필요 — 응답 길이, 톤, 답변 형식 조정 필요

**박서연(AI 엔지니어):** 원인 A는 당장 고쳐도 전체 오답의 절반 정도는 해결될 거라 봅니다. 나머지 절반은 B와 C의 조합 문제입니다.

**최민준(백엔드):** B 문제를 해결하려면 청킹 방식을 바꿔야 하는데, 이 경우 지식 베이스 전체를 다시 임베딩해야 합니다. 작업 부담이 크지 않은 건 아닌데, 어차피 문서 갱신을 하면서 같이 하면 됩니다.

## 1. 이전 액션 이행 확인 및 공유

**박서연:** 지식 베이스 문서는 최신 정책 기준으로 갱신 완료했습니다. semantic chunking 방식으로는 LangChain의 RecursiveCharacterTextSplitter를 활용하는 방안과, 문장 임베딩 기반으로 유사도 임계값을 두어 청크를 합치는 방식 두 가지를 검토했습니다. 두 번째 방법이 정책 문서처럼 조건문이 많은 텍스트에 더 효과적이라 판단하고 있습니다.

**최민준:** 재임베딩 파이프라인은 준비되어 있습니다. 청킹 방식만 확정되면 바로 실행 가능합니다.

**윤재혁:** QA 케이스 30개 준비 완료했습니다. 반품(10개), 환불(8개), 배송(6개), 회원/포인트(6개)로 구성했습니다.

## 2. 개선 방안 상세 논의

### 2-1. 파인튜닝 도입 가능성 검토

**박서연(AI 엔지니어):** 이번 개선 방향을 논의하면서 파인튜닝 옵션도 검토해봤습니다. 파인튜닝을 하면 CS 응답 톤과 도메인 특화 표현을 모델 수준에서 학습시킬 수 있어서 응답 품질이 구조적으로 개선됩니다. 그린커머스의 CS 대화 데이터를 학습 데이터로 쓸 수 있다면 효과적입니다.

**이지훈(PM):** 현실적으로 가능한지 먼저 검토해봅시다. 파인튜닝에 필요한 게 무엇인지 정리해주시겠어요?

**박서연(AI 엔지니어):** 최소 500~1000쌍의 질의-답변 학습 데이터가 필요합니다. 데이터 품질이 중요해서 단순히 양이 많다고 되는 게 아니고 정제 과정이 필요합니다. 비용 면에서는 GPT-4o 파인튜닝 기준으로 데이터 준비 포함 최소 2~3주 이상 소요됩니다. 그 외에 반복 실험과 검증 비용도 있습니다.

**윤재혁(QA):** 고객사에서 제공받은 로그 데이터의 품질이 어떤가요? 그걸 학습 데이터로 쓸 수 있는지가 관건일 것 같습니다.

**최민준(백엔드):** 로그 데이터를 보면 답변이 틀린 케이스가 많습니다. 그 데이터를 그대로 쓰면 잘못된 패턴을 학습할 수 있어서 사전 라벨링과 필터링 작업이 반드시 필요합니다.

**강다은(프론트엔드):** 지금 일정상 가능한가요? 고객사가 빠른 개선을 원하고 있는데, 파인튜닝에 3주 이상 쓰면 다른 개선은 손도 못 댑니다.

**이지훈(PM):** 현재 우리 팀 리소스 상황을 보면, 박서연씨는 RAG 파이프라인 개선과 프롬프트 재설계를 병행하고 있고, 최민준씨는 재임베딩 파이프라인과 서버 안정화를 맡고 있습니다. 파인튜닝을 위한 데이터 준비와 학습 실험을 추가로 맡을 여유가 없는 상황입니다. 게다가 고객사가 2주 내 1차 개선 결과를 요청한 상태입니다.

---

**[결정 1]** 파인튜닝은 이번 유지보수에서 제외하기로 결정했다.  
이유: 학습 데이터 준비에 최소 2주 이상이 필요하고, 데이터 라벨링 및 정제 공수가 크며, 현재 팀 가용 리소스로는 2주 내 1차 개선 목표를 달성하면서 병행하기 불가능하다. 파인튜닝은 추후 2차 유지보수 또는 계약 연장 시 별도 과제로 검토한다.

---

### 2-2. RAG 파이프라인 개선 방향 확정

**박서연:** 파인튜닝 대신 RAG 품질 개선에 집중하는 방향을 제안합니다. 구체적으로는 세 가지입니다. 첫째, 앞서 논의한 semantic chunking 적용. 둘째, 벡터 검색에 BM25 키워드 검색을 결합한 하이브리드 검색 도입. 셋째, 검색 결과 상위 N개를 re-ranking하여 관련도 높은 청크를 LLM에 전달.

**최민준:** BM25 하이브리드 도입은 기존 구조에서 크게 변경 없이 추가 가능합니다. re-ranking은 cross-encoder 모델을 쓰면 응답 지연이 조금 늘어날 수 있는데, 어느 정도가 허용 범위인지 고객사와 확인이 필요합니다.

**이지훈:** 응답 지연 기준을 고객사에 확인하겠습니다. 현재 기준에서 얼마나 늘어날 것 같습니까?

**최민준:** cross-encoder 추가 시 약 0.5~1초 정도 추가될 것으로 예상합니다. 현재 평균 응답 시간이 2.3초이니, 3~3.5초 수준이 됩니다.

**박서연:** re-ranking의 효과가 크다면 이 정도 지연은 감수할 만합니다. 다만 QA 테스트를 통해 검색 품질 개선 효과를 확인한 뒤 최종 도입 여부를 결정하는 것이 좋겠습니다.

---

**[결정 2]** RAG 파이프라인 개선은 semantic chunking + 하이브리드 검색(벡터 + BM25)을 우선 적용하고, re-ranking은 QA 검증 결과를 보고 도입 여부를 결정하기로 했다.  
이유: re-ranking은 응답 지연 가능성이 있어 먼저 검증이 필요하다.

---

### 2-3. LLM 모델 최신화 논의

**박서연:** 납품 당시 사용한 모델 버전이 오래되었습니다. 현재 OpenAI API를 통해 호출 중인데, 최근 출시된 모델들이 응답 품질 면에서 더 우수하다는 게 업계 평가입니다. 모델 업데이트도 이번 유지보수에 포함하는 것을 제안합니다.

**이지훈:** 모델 교체 시 비용은 어떻게 됩니까?

**박서연:** 현재 사용 중인 모델보다 최신 모델은 토큰당 단가가 다를 수 있습니다. 현재 그린커머스 기준 월 약 340건/일 × 30일 = 10,200건 수준이고, 평균 입출력 토큰을 계산하면 비용 차이는 월 3~5만 원 수준으로 예상합니다. 큰 부담은 아니지만 고객사 계약에 운영 비용 조정이 필요할 수 있습니다.

**최민준:** 모델만 바꾸는 건 코드 변경이 크지 않습니다. API 엔드포인트와 모델명 파라미터만 수정하면 됩니다. 다만 새 모델에서 프롬프트 동작이 달라질 수 있어서 프롬프트 재테스트가 필요합니다.

---

**[결정 3]** LLM 모델을 최신 버전으로 업데이트하기로 결정했다.  
이유: 현재 사용 모델이 납품 시점 이후 업데이트되지 않았으며, 최신 모델의 응답 품질이 개선된 것이 확인되었다. 비용 증가 폭이 크지 않아 실용적 판단이 가능하다. 단, 모델 교체 후 프롬프트 재테스트를 반드시 진행한다.

## 1. 1차 개선 작업 이행 확인

**박서연:** Semantic chunking 적용 완료했습니다. 정책 문서의 조건 구조가 이전보다 훨씬 잘 보존됩니다. LLM 모델도 최신 버전으로 업데이트했고, 프롬프트는 세 가지 버전을 작성해서 테스트 중입니다.

**최민준:** 재임베딩 완료했고 하이브리드 검색 모듈 연동도 완료했습니다. 응답 시간은 평균 1.9초로 오히려 이전보다 개선됐습니다. 청크가 더 의미 단위로 잘 잘리니까 LLM 처리 부담이 줄어든 것 같습니다.

## 2. QA 결과 보고

**윤재혁(QA):**

30개 케이스 테스트 결과를 보고합니다.

- 전체 정답률: 이전 55% → 현재 **80%**
- 반품 관련(10개): 이전 55% → 현재 **90%** (정책 갱신 효과가 가장 큼)
- 환불 관련(8개): 이전 50% → 현재 **75%**
- 배송 관련(6개): 이전 67% → 현재 **83%**
- 회원/포인트(6개): 이전 50% → 현재 **67%**

가장 크게 개선된 영역은 반품이었고, 예상대로 정책 문서 갱신과 semantic chunking 효과가 컸습니다. 회원/포인트 영역은 아직 오답이 많은데, 이 쪽 문서가 충분하지 않은 것 같습니다.

**이지훈:** 전체 80%는 의미 있는 개선입니다. 고객사가 제시한 기준이 있나요?

**이지훈:** 지난주 고객사에 문의했을 때 "체감상 크게 좋아졌으면 한다"고 했는데 수치 기준은 별도로 제시하지 않았습니다. 다만 계약 연장을 검토하려면 "눈에 띄는 개선"이 필요하다고 했습니다.

**박서연:** 80%도 좋지만 회원/포인트 영역의 67%는 여전히 낮습니다. 이 영역 지식 베이스를 보강하면 전체 수치가 더 올라갈 것 같습니다.

## 3. re-ranking 도입 논의

**박서연:** re-ranking을 추가로 적용했을 때 어떻게 달라지는지 예비 테스트를 해봤습니다. cross-encoder 기반으로 상위 10개 검색 결과를 재정렬했더니 환불 영역 정확도가 75%에서 88%로 올랐습니다. 응답 시간 증가는 평균 0.6초였습니다.

**최민준:** 0.6초 추가면 평균 2.5초 수준이 되는데, 허용 가능한 범위입니다.

**윤재혁:** 추가 테스트 없이 전체 도입해도 될 만한 수치입니다.

---

**[결정 1]** re-ranking(cross-encoder 기반)을 정식 도입하기로 결정했다.  
이유: 예비 테스트에서 환불 영역 정확도가 13%p 향상됐고, 응답 지연 증가(+0.6초)가 허용 가능한 수준으로 확인됐다.

## 5. 멀티턴 맥락 유지 기능 논의

**강다은:** 지난주 조사 결과를 공유합니다. 현재 아키텍처에서 멀티턴을 구현하려면 세션별 대화 이력을 서버에서 관리하고, 각 질문에 이전 대화 컨텍스트를 함께 LLM에 전달하는 방식이 필요합니다. 구현 공수는 백엔드와 프론트 합산 약 3~4일 정도로 봅니다.

**최민준:** 컨텍스트를 많이 넣을수록 토큰 비용이 늘어납니다. 대화 이력 길이에 상한을 두는 것이 좋습니다.

**이지훈:** 이 기능은 고객사에서 명시적으로 요청한 게 아닙니다. 그러나 프론트 로그를 보면 같은 주제로 반복 질문하는 패턴이 있었고, 이 기능이 있으면 사용자 경험이 크게 좋아질 것 같습니다.

**박서연:** 지금 일정에서는 2차 배포 때 포함하는 게 현실적입니다. 1차 배포 일정에 무리하게 끼워 넣으면 품질 리스크가 있습니다.

---

**[결정 3]** 멀티턴 맥락 유지 기능은 2차 배포 항목으로 분리하여 다음 스프린트에 포함하기로 결정했다.

## 1. 최종 QA 결과 보고

**윤재혁(QA):**

re-ranking 통합 및 회원/포인트 문서 보강 이후 최종 30개 케이스 재테스트를 진행했습니다.

| 영역 | 1차 개선 전 | 1차 개선 후 | 최종 |
|------|------------|------------|------|
| 반품 (10개) | 55% | 90% | **90%** |
| 환불 (8개) | 50% | 75% | **88%** |
| 배송 (6개) | 67% | 83% | **83%** |
| 회원/포인트 (6개) | 50% | 67% | **83%** |
| **전체** | **55%** | **80%** | **87%** |

전체 정답률이 87%까지 올라왔습니다. 회원/포인트 영역은 문서 보강 효과가 컸고, 환불 영역은 re-ranking 도입 효과가 명확했습니다.

**이지훈:** 55%에서 87%면 32%p 개선입니다. 충분히 "눈에 띄는 개선"이라 할 수 있을 것 같습니다.

**박서연:** 한 가지 추가로 말씀드릴 점이 있습니다. 테스트 중에 할루시네이션이 발생한 케이스가 2건 있었는데, 모두 지식 베이스에 관련 문서가 없는 질문이었습니다. 문서가 없는 경우 명시적으로 "확인이 어렵습니다"라고 답하도록 프롬프트에 fallback 가이드라인을 추가했습니다.

**윤재혁:** 이 fallback 처리가 잘 작동하고 있는지 배포 이후에도 모니터링이 필요합니다.

## 2. 배포 일정 확정

**이지훈:** 고객사와 6/28 사전 보고를 마쳤습니다. 테스트 결과를 공유했더니 긍정적인 반응이었습니다. 이번 주말 내 1차 배포를 완료하자는 제안을 드렸고, 고객사 측에서 7/1(수) 이전까지 배포해달라고 요청했습니다.

**최민준:** 배포 자체는 준비되어 있습니다. 기존 서버 환경에 그대로 올리면 되는데, 재임베딩된 벡터 DB와 업데이트된 LLM 호출 로직, re-ranking 모듈을 한 번에 올려야 합니다. 배포 전 스테이징 환경에서 30분 정도 최종 확인이 필요합니다.

**박서연:** 배포 후 모델 동작이 프로덕션에서 다르게 나올 수도 있어서 24시간 모니터링 체계가 필요합니다.
