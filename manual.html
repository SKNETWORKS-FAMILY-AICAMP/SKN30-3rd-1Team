1. 구성 개요

권장 운영 방식은 쓰기 에이전트 하나와 읽기 전용 검증 역할 둘을 순차 실행하는 구조입니다.

사용자 Task 입력 ↓ Claude Code: 현재 변경사항 분석 및 계획 작성 ↓ 사용자: 계획 승인 ↓ Claude Code: 백엔드 구현·테스트 ↓ 결정론적 검사: scope / lint / type / test / build ↓ Codex Reviewer: 독립 코드 리뷰 ↓ Claude Code: 승인된 리뷰 항목 수정 ↓ Codex Validator 새 세션: 완료 조건 검증 ↓ 사용자: 최종 승인 ↓ 완료
핵심: 에이전트의 “완료했습니다”라는 문장은 완료 근거가 아닙니다. 테스트와 정적 검사, Validator의 PASS, 사용자 최종 승인이 모두 필요합니다.

사용자가 직접 승인하는 시점

승인확인 내용이후 동작
계획 승인변경 범위, 수정 파일, 테스트 계획, 리스크Claude 구현 시작
최종 승인최종 diff, 테스트 결과, 리뷰 해결 현황, 잔여 리스크작업 완료 처리

주의: 모델 이름과 사용 가능 여부는 계정, Claude Code/Codex 버전 및 시점에 따라 달라집니다. 실제 CLI에 표시되는 목록을 우선합니다. 본 표는 2026-07-10 기준입니다.
Ultra 모드 금지: GPT-5.6의 ultra는 서브에이전트를 병렬 실행합니다. read-only 리뷰의 결정론성을 해치므로 이 워크플로우에서는 사용하지 않습니다. 또한 xhigh는 GPT-5.5 시절 명칭입니다. GPT-5.6의 단계는 Low / Medium / High / Extra High이며, 그 위에 Max(단일 에이전트 심화)와 Ultra(병렬)가 별도로 존재합니다. GPT-5.5의 effort 값은 GPT-5.6에 그대로 매핑되지 않으므로, 익숙한 작업을 낮은 설정으로 먼저 돌려보고 조정하세요.
Rocky 8 / RHEL 계열 + WSL2 사전 준비: Codex의 리눅스 샌드박스는 bubblewrap을 씁니다. sudo dnf install -y bubblewrap으로 설치하세요. RHEL 계열은 user.max_user_namespaces가 기본 0이라 unprivileged user namespace 생성이 막혀 있습니다(DISA STIG 요구사항이기도 합니다). 이 경우 두 가지 선택지가 있습니다: sudo chmod u+s $(command -v bwrap)로 bwrap에 setuid를 주거나(권장, 시스템 전역 설정 불변), user.max_user_namespaces를 0이 아닌 값으로 바꾸는 것(전역 완화이므로 보안 정책 확인 필요)입니다. 커널은 WSL2의 Microsoft 커널이 쓰이므로 Rocky 8의 4.18이 아닙니다 — Landlock에 필요한 5.13 이상인지 uname -r로 확인하고, 낮으면 Windows PowerShell에서 wsl --update를 실행하세요. ./scripts/sandbox-probe.sh가 이 모든 항목을 자동 점검합니다.
검증된 호출 형태 (Codex 0.144 기준):
codex exec -s read-only -m <model> -C . --color never -- "프롬프트"
codex exec에는 -a/--ask-for-approval이 없습니다. exec은 본래 비대화형이라 승인 개념이 없습니다. --cd의 단축은 -C이고 -c는 --config인 별개 플래그입니다. 프롬프트는 -- 뒤에 두세요. 그렇지 않으면 서브커맨드로 오인될 수 있습니다. 저장소 밖에서 돌리려면 --skip-git-repo-check가 필요합니다. 플래그는 버전마다 바뀝니다. 업그레이드 후 codex exec --help로 확인하고 ./scripts/sandbox-probe.sh를 다시 돌리세요.
read-only는 프롬프트로 보장되지 않습니다. Codex를 반드시 --sandbox read-only 또는 승인 게이트 모드로 구동하세요. GPT-5.6 Sol은 GPT-5.5보다 파일 수정·셸 실행 같은 행동을 더 적극적으로 취합니다. "Do not edit"이라고 쓰는 것만으로는 통제가 되지 않습니다.
Fable 5 fallback: Fable 5에는 안전 분류기가 있어 사이버보안·생물학 인접 요청이 Opus 4.8로 라우팅될 수 있습니다. 저장소에 인증·암호화·취약점 스캐너 코드가 있으면 세션이 전환될 수 있습니다. Claude Code v2.1.170 이상에서 /model fable로 선택합니다. API 사용 시 거부는 HTTP 200에 stop_reason: "refusal"로 반환됩니다.

권장 운용

  • 평범한 백엔드 기능·버그 수정: Sonnet 5 → GPT-5.6 Sol high → GPT-5.6 Terra medium
  • 복잡한 DB·동시성·아키텍처 작업: Fable 5 → GPT-5.6 Sol max → GPT-5.6 Sol medium
  • Validator는 Reviewer와 반드시 새 세션으로 실행합니다.
  • macOS x86_64 + Codex CLI 0.142.5에서 gpt-5.6-sol이 셸 호출 시 SIGTRAP(exit 133)로 종료되는 이슈가 있습니다. Linux/WSL/ARM macOS는 영향 없습니다.

3. 생성된 파일 구조

AGENTS.md Codex 영문 지침 CLAUDE.md Claude Code 영문 지침 README.ko.md 빠른 시작 안내 manual.html 이 매뉴얼 docs/ ├─ backend-rules.md 공통 백엔드 규칙(영문) ├─ review-rules.md 리뷰 판정 규칙(영문) └─ validation-rules.md 최종 검증 규칙(영문) .agent-workflow/ ├─ config.yaml 워크플로 정책(사람이 읽음) ├─ config.sh 경로·검사 명령(스크립트가 읽음, 단일 소스) ├─ templates/ 한글 작업 문서 템플릿 └─ tasks/ Task별 기록 저장 scripts/ ├─ agent-task.sh Task 생성·상태·승인 ├─ agent-task.ps1 PowerShell 진입 래퍼 ├─ check-scope.sh 프론트엔드 변경 차단 └─ verify-backend.sh 백엔드 검증 실행

영문 에이전트가 반복해서 읽는 지침과 규칙, 설정. 한글 사용자가 검토하는 계획·보고서·대시보드·검증 결과.

4. 최초 설치

패키지 파일 복사

압축을 프로젝트 루트에 풀어 파일을 병합합니다. 기존 AGENTS.md 또는 CLAUDE.md가 있다면 덮어쓰지 말고 내용을 병합합니다.

검증 명령 설정

.agent-workflow/config.sh의 CMD_* 변수를 실제 프로젝트에 맞게 작성합니다.

commands:
  format_check: "ruff format --check backend tests"
  lint: "ruff check backend tests"
  typecheck: "mypy backend"
  test: "pytest -q"
  build: ""
  smoke_test: "python -c 'import backend'"

예시는 Python 프로젝트입니다. 빈 명령은 SKIP 처리됩니다. 프로젝트 구조를 확인한 뒤 정확한 명령으로 교체해야 합니다.

백엔드·금지 경로 설정

실제 폴더명이 다르면 .agent-workflow/config.sh의 FORBIDDEN_PATHS를 수정합니다. 스크립트는 이 배열만 읽으므로 정규식을 따로 손댈 필요가 없습니다. config.yaml의 목록은 사람이 읽기 위한 사본입니다.

실행 권한

chmod +x scripts/*.sh

Windows에서는 WSL 또는 Git Bash 사용을 권장합니다. PowerShell에서는 agent-task.ps1이 내부적으로 Bash를 호출합니다.

5. 기존 로컬 변경사항 보존

현재 수정만 진행된 상태이므로, 먼저 작업 시작 전 상태를 개인 브랜치의 기준 커밋으로 고정합니다.

먼저 확인: 팀원의 작업이나 비밀값이 포함되어 있지 않은지 git diff로 직접 확인하세요.
# 1. 무엇이 커밋될지 먼저 본다 (untracked 포함)
git status --short
git diff
git ls-files --others --exclude-standard   # 새 파일 목록

# 2. .gitignore에 비밀값 패턴이 있는지 확인
cp .gitignore.example .gitignore   # 없다면
grep -E '\.env|\.pem|\.key' .gitignore

# 3. 현재 로컬 변경을 유지한 채 개인 브랜치 생성
git switch -c develop/my-work

# 4. 체크포인트 커밋
git add -A
git status --short   # 스테이징 결과 재확인. .env 등이 없어야 한다
git commit -m "wip: checkpoint existing backend changes"

# 5. baseline을 태그로 고정 (커밋 해시보다 다루기 쉽다)
git tag agent-baseline/TASK-001
git rev-parse HEAD
git add -A는 눈으로 확인한 뒤에만 실행하세요. 이 명령은 .env, 로컬 인증서, 팀원의 미완성 작업까지 전부 담습니다. 2단계를 건너뛰지 마세요. scripts/check-scope.sh가 비밀값 파일을 잡아내지만, 그건 이미 커밋된 뒤에는 늦습니다.

이 체크포인트는 완성 커밋이 아닙니다. 나중에 팀에 반영할 때 squash 또는 정리할 수 있습니다.

결과: main은 공동 기준으로 남고, 모든 AI 작업은 develop/my-work에서 진행됩니다.

6. Task 생성

./scripts/agent-task.sh init TASK-001 "현재 백엔드 변경사항 검토 및 보완" HEAD
./scripts/agent-task.sh status TASK-001

PowerShell 예:

.\scripts\agent-task.ps1 -Command init -TaskId TASK-001 `
  -Title "현재 백엔드 변경사항 검토 및 보완" -Baseline HEAD

생성 후 다음 두 파일을 사람이 먼저 작성합니다.

  • task.md: 작업 목적, 현재 변경사항, 변경 가능·금지 범위
  • acceptance-criteria.md: 기능·예외·테스트 기준
Task 문서가 모호하면 모든 후속 판단도 모호해집니다. 완료 조건은 가능한 한 “관찰 가능한 결과”로 씁니다.

7. Claude 계획 작성 및 승인

Claude Code에 아래 프롬프트를 전달합니다. 이 단계에서는 소스 수정이 금지됩니다.

Read CLAUDE.md and all files under .agent-workflow/tasks/TASK-001/.
Analyze the current local backend changes and the baseline-to-current diff.
Do not modify source code yet.
Write the implementation plan in Korean to:
.agent-workflow/tasks/TASK-001/plan.md
Stop after the plan is written and wait for user approval.

사용자는 plan.md에서 수정 예정 파일, 수정 금지 파일, 테스트 계획, 리스크를 확인합니다.

./scripts/agent-task.sh approve-plan TASK-001
./scripts/agent-task.sh status TASK-001

8. Claude 구현 및 사전 검사

The plan for TASK-001 is approved.
Implement only the approved backend scope.
Preserve unrelated local changes and do not modify frontend files.
Add regression tests, run scripts/verify-backend.sh, and write the Korean
implementation report to .agent-workflow/tasks/TASK-001/implementation-report.md.
Do not commit, push, merge, or declare completion.

범위 검사를 활성화하려면 기준 커밋을 환경변수로 넘깁니다.

export BASELINE_COMMIT=$(python3 -c "import json; print(json.load(open('.agent-workflow/tasks/TASK-001/state.json'))['baseline_commit'])")
./scripts/verify-backend.sh 2>&1 | tee .agent-workflow/tasks/TASK-001/logs/precheck.log
검사 명령이 설정되지 않아 SKIP된 항목이 있으면 아직 검증 완료가 아닙니다. config.yaml을 먼저 보완하세요.

9. Codex 독립 리뷰

Codex를 반드시 read-only 샌드박스 플래그로 구동하고, Claude의 대화 기록이 없는 새 세션을 사용합니다. 프롬프트의 "Do not edit"은 통제 수단이 아닙니다.

Act as the read-only backend reviewer for TASK-001.
Read AGENTS.md, docs/review-rules.md, and the task files.
Review the baseline-to-current Git diff and directly related code paths.
Do not edit any source file.
Write the review in Korean to:
.agent-workflow/tasks/TASK-001/review.md
Report only concrete defects and missing acceptance criteria.

리뷰 후 review-triage.md에서 각 항목을 판정합니다.

판정의미
ACCEPTED코드 근거가 있고 수정 대상임
REJECTED오탐, 범위 밖, 또는 설계상 의도된 동작
NEEDS_HUMAN요구사항·아키텍처 판단이 필요한 항목

10. Claude 리뷰 반영

Read the TASK-001 review and review triage.
Fix only findings marked ACCEPTED.
Do not implement REJECTED or NEEDS_HUMAN findings.
Add regression tests where feasible, run backend verification, and write the
Korean resolution report to:
.agent-workflow/tasks/TASK-001/review-resolution.md
Do not commit or declare completion.

자동 리뷰·수정은 최대 2회가 기본값입니다. 2회 이후에도 High 이상 이슈가 남으면 사람에게 넘깁니다.

11. Codex 최종 검증

Reviewer와 별개의 새 Codex 세션에서 수행합니다. 검증은 새로운 개선점을 무한히 찾는 단계가 아니라, 완료 조건 충족 여부를 확인하는 단계입니다.

Act as the read-only backend validator for TASK-001 in a fresh session.
Read AGENTS.md, docs/validation-rules.md, acceptance-criteria.md, review.md,
review-triage.md, review-resolution.md, the verification logs, and the final diff.
Re-run deterministic checks. Do not edit source files.
Write the Korean result to .agent-workflow/tasks/TASK-001/validation.md.
The final status must be exactly PASS, FAIL, or BLOCKED.

PASS 완료 조건 충족. FAIL 코드 수정으로 해결 가능. BLOCKED 환경·요구사항·권한이 없어 판단 불가.

PASS 후 사용자가 최종 diff와 리스크를 확인합니다.

# Validator 결과를 먼저 기록한다
./scripts/agent-task.sh set-validation TASK-001 PASS

# 최종 diff 확인
git status --short
git diff agent-baseline/TASK-001..HEAD
git log --oneline agent-baseline/TASK-001..HEAD

# PASS가 기록되어 있지 않으면 아래 명령은 거부된다
./scripts/agent-task.sh approve-final TASK-001
강제됨: approve-final은 validation_status가 PASS가 아니면 실패합니다. "완료했습니다"라는 문장으로는 이 게이트를 통과할 수 없습니다.

13. 중단·실패 대응

상황대응
Claude가 프론트엔드를 수정함자동으로 폐기하지 말고 diff를 확인한 뒤 해당 변경만 수동 복원합니다. Validator는 FAIL 처리합니다.
테스트 환경이 없음검증 결과를 BLOCKED로 남기고 필요한 환경·서비스를 명시합니다.
Codex 리뷰가 과도함review-triage.md에서 REJECTED 처리하고 근거를 기록합니다.
기존 변경사항이 훼손됨기준 커밋과 현재 diff를 비교합니다. 즉시 reset하지 말고 별도 patch를 먼저 보존합니다.
리뷰 2회를 넘겨야 함review-round가 거부합니다. 남은 High 이상 항목은 사람이 판단합니다. 한도를 늘리려면 config.sh의 MAX_REVIEW_ROUNDS를 수정하되, 이유를 기록하세요.
검증이 INCONCLUSIVE로 나옴config.sh에 검사 명령이 하나도 없습니다. 아무것도 검증되지 않은 상태이며 PASS가 아닙니다. 명령을 채우고 다시 실행하세요.
Critical 항목이 NEEDS_HUMAN으로 남음Validator는 BLOCKED를 반환합니다. review-triage.md에 사람의 결정을 기록해야 진행됩니다.
작업을 취소하고 싶음현재 diff를 patch로 저장한 후 개인 브랜치를 정리합니다. 공동 main에는 영향을 주지 않습니다.
# 안전한 임시 보존
git diff > ../TASK-001-working.patch
git status

# 기준 변경사항 확인
git show <baseline-commit>
git diff <baseline-commit>
자동화 지침은 git reset --hard, git clean -fd, 강제 push를 금지합니다. 복구 전에는 항상 patch 또는 별도 브랜치로 보존하세요.
