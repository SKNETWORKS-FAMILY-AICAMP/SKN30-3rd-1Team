## 1. 이전 액션 이행 확인 및 공유

**박서연:** 지식 베이스 문서는 최신 정책 기준으로 갱신 완료했습니다. semantic chunking 방식으로는 LangChain의 RecursiveCharacterTextSplitter를 활용하는 방안과, 문장 임베딩 기반으로 유사도 임계값을 두어 청크를 합치는 방식 두 가지를 검토했습니다. 두 번째 방법이 정책 문서처럼 조건문이 많은 텍스트에 더 효과적이라 판단하고 있습니다.

**최민준:** 재임베딩 파이프라인은 준비되어 있습니다. 청킹 방식만 확정되면 바로 실행 가능합니다.

## 2. 개선 방안 상세 논의

### 2-1. 파인튜닝 도입 가능성 검토

**박서연(AI 엔지니어):** 이번 개선 방향을 논의하면서 파인튜닝 옵션도 검토해봤습니다. 파인튜닝을 하면 CS 응답 톤과 도메인 특화 표현을 모델 수준에서 학습시킬 수 있어서 응답 품질이 구조적으로 개선됩니다. 그린커머스의 CS 대화 데이터를 학습 데이터로 쓸 수 있다면 효과적입니다.

**이지훈(PM):** 현실적으로 가능한지 먼저 검토해봅시다. 파인튜닝에 필요한 게 무엇인지 정리해주시겠어요?

**박서연(AI 엔지니어):** 최소 500~1000쌍의 질의-답변 학습 데이터가 필요합니다. 데이터 품질이 중요해서 단순히 양이 많다고 되는 게 아니고 정제 과정이 필요합니다. 비용 면에서는 GPT-4o 파인튜닝 기준으로 데이터 준비 포함 최소 2~3주 이상 소요됩니다. 그 외에 반복 실험과 검증 비용도 있습니다.

**윤재혁(QA):** 고객사에서 제공받은 로그 데이터의 품질이 어떤가요? 그걸 학습 데이터로 쓸 수 있는지가 관건일 것 같습니다.

**최민준(백엔드):** 로그 데이터를 보면 답변이 틀린 케이스가 많습니다. 그 데이터를 그대로 쓰면 잘못된 패턴을 학습할 수 있어서 사전 라벨링과 필터링 작업이 반드시 필요합니다.

**강다은(프론트엔드):** 지금 일정상 가능한가요? 고객사가 빠른 개선을 원하고 있는데, 파인튜닝에 3주 이상 쓰면 다른 개선은 손도 못 댑니다.

**이지훈(PM):** 현재 우리 팀 리소스 상황을 보면, 박서연씨는 RAG 파이프라인 개선과 프롬프트 재설계를 병행하고 있고, 최민준씨는 재임베딩 파이프라인과 서버 안정화를 맡고 있습니다. 파인튜닝을 위한 데이터 준비와 학습 실험을 추가로 맡을 여유가 없는 상황입니다. 게다가 고객사가 2주 내 1차 개선 결과를 요청한 상태입니다.

---

**[결정 1]** 파인튜닝은 이번 유지보수에서 제외하기로 결정했다.  
이유: 학습 데이터 준비에 최소 2주 이상이 필요하고, 데이터 라벨링 및 정제 공수가 크며, 현재 팀 가용 리소스로는 2주 내 1차 개선 목표를 달성하면서 병행하기 불가능하다. 파인튜닝은 추후 2차 유지보수 또는 계약 연장 시 별도 과제로 검토한다.

---

### 2-2. RAG 파이프라인 개선 방향 확정

**박서연:** 파인튜닝 대신 RAG 품질 개선에 집중하는 방향을 제안합니다. 구체적으로는 세 가지입니다. 첫째, 앞서 논의한 semantic chunking 적용. 둘째, 벡터 검색에 BM25 키워드 검색을 결합한 하이브리드 검색 도입. 셋째, 검색 결과 상위 N개를 re-ranking하여 관련도 높은 청크를 LLM에 전달.

**최민준:** BM25 하이브리드 도입은 기존 구조에서 크게 변경 없이 추가 가능합니다. re-ranking은 cross-encoder 모델을 쓰면 응답 지연이 조금 늘어날 수 있는데, 어느 정도가 허용 범위인지 고객사와 확인이 필요합니다.

**이지훈:** 응답 지연 기준을 고객사에 확인하겠습니다. 현재 기준에서 얼마나 늘어날 것 같습니까?

**최민준:** cross-encoder 추가 시 약 0.5~1초 정도 추가될 것으로 예상합니다. 현재 평균 응답 시간이 2.3초이니, 3~3.5초 수준이 됩니다.

**박서연:** re-ranking의 효과가 크다면 이 정도 지연은 감수할 만합니다. 다만 QA 테스트를 통해 검색 품질 개선 효과를 확인한 뒤 최종 도입 여부를 결정하는 것이 좋겠습니다.

---

**[결정 2]** RAG 파이프라인 개선은 semantic chunking + 하이브리드 검색(벡터 + BM25)을 우선 적용하고, re-ranking은 QA 검증 결과를 보고 도입 여부를 결정하기로 했다.  
이유: re-ranking은 응답 지연 가능성이 있어 먼저 검증이 필요하다.

---

### 2-3. LLM 모델 최신화 논의

**박서연:** 납품 당시 사용한 모델 버전이 오래되었습니다. 현재 OpenAI API를 통해 호출 중인데, 최근 출시된 모델들이 응답 품질 면에서 더 우수하다는 게 업계 평가입니다. 모델 업데이트도 이번 유지보수에 포함하는 것을 제안합니다.

**이지훈:** 모델 교체 시 비용은 어떻게 됩니까?

**박서연:** 현재 사용 중인 모델보다 최신 모델은 토큰당 단가가 다를 수 있습니다. 현재 그린커머스 기준 월 약 340건/일 × 30일 = 10,200건 수준이고, 평균 입출력 토큰을 계산하면 비용 차이는 월 3~5만 원 수준으로 예상합니다. 큰 부담은 아니지만 고객사 계약에 운영 비용 조정이 필요할 수 있습니다.

**최민준:** 모델만 바꾸는 건 코드 변경이 크지 않습니다. API 엔드포인트와 모델명 파라미터만 수정하면 됩니다. 다만 새 모델에서 프롬프트 동작이 달라질 수 있어서 프롬프트 재테스트가 필요합니다.

---

**[결정 3]** LLM 모델을 최신 버전으로 업데이트하기로 결정했다.  
이유: 현재 사용 모델이 납품 시점 이후 업데이트되지 않았으며, 최신 모델의 응답 품질이 개선된 것이 확인되었다. 비용 증가 폭이 크지 않아 실용적 판단이 가능하다. 단, 모델 교체 후 프롬프트 재테스트를 반드시 진행한다.

## 3. 1차 개선 범위 최종 확정

이번 유지보수 1차 개선 범위는 다음과 같이 확정한다.

1. 지식 베이스 문서 최신화 (완료)
2. Semantic chunking 도입 + 전체 재임베딩
3. 하이브리드 검색(벡터 + BM25) 적용
4. 프롬프트 전면 재설계
5. LLM 모델 최신 버전 업데이트

재임베딩 후 QA 테스트 결과에 따라 re-ranking 추가 여부 결정.
