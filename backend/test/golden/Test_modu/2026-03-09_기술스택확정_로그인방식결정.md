## 1. 지난 액션 이행 확인

박현우(프론트/앱): 앱 프레임워크 조사 결과, Flutter를 추천합니다. iOS/Android 동시 개발이 가능해서 10주 일정에 맞고, 우리 팀 러닝커브도 크지 않습니다.

이수진(백엔드): 서버는 Node.js + PostgreSQL로 제안합니다. 위치 기반 쿼리가 많을 텐데 PostGIS 확장을 쓰면 500m 반경 검색을 효율적으로 처리할 수 있습니다.

한지민(QA·마케팅): 경쟁 앱 분석 결과, 대부분 이메일과 소셜 로그인을 함께 제공하고 있었습니다. 특히 신규 앱일수록 가입 이탈을 줄이려고 소셜 로그인 비중을 높이는 추세입니다.

## 2. 로그인 방식 논의

김태호(PM): 오늘 핵심 안건은 로그인 방식입니다. 어떤 방식으로 갈지 정해야 백엔드가 인증 설계를 시작할 수 있습니다.

이수진(백엔드): 저는 이메일 + 비밀번호 방식을 기본으로 제안합니다. 구현이 단순하고 외부 의존성이 없어서 안정적입니다. 소셜 로그인은 각 플랫폼 심사와 SDK 연동이 필요해서 초기에는 부담입니다.

박현우(프론트/앱): 동의합니다. 이메일 방식이 가장 빠르게 붙일 수 있습니다.

한지민(QA·마케팅): 다만 리서치 결과를 보면, 20~30대 타깃은 이메일 가입을 번거로워하는 경향이 강합니다. 이탈률이 우려됩니다.

김태호(PM): 일단 MVP는 속도가 중요하니 이메일 + 비밀번호로 시작합시다. 소셜 로그인은 출시 후 추가하는 걸로 하죠.

---

## 3. 결정사항

**[결정 1]** 앱 개발 프레임워크를 Flutter로 확정한다.
이유: iOS/Android 동시 개발로 일정 단축 가능, 팀 러닝커브 낮음.

**[결정 2]** 백엔드를 Node.js + PostgreSQL(PostGIS)로 확정한다.
이유: 위치 기반 반경 검색에 PostGIS가 적합.

**[결정 3]** 로그인 방식을 이메일 + 비밀번호 방식으로 확정한다. 소셜 로그인은 출시 후 2차 업데이트로 미룬다.
이유: MVP 개발 속도를 우선하며, 외부 플랫폼 심사·SDK 연동 부담을 초기에 지지 않기 위함.

**[결정 4]** 채팅 기능은 실시간(WebSocket) 대신 비실시간 게시판형 댓글로 MVP에 구현한다.
이유: 실시간 채팅 구현 공수가 크고, MVP 단계에서는 모임 내 소통이 가능한 수준이면 충분.
