# 중간 회의록

날짜: 2026-06-30
참석자: 김민수, 박지현, 이서연, 최도윤

## 회의 목적

PaiM 문서 업로드 흐름에서 사용자가 기다리는 시간을 줄이고, 추출된 프로젝트 메모리의 정확도를 높이는 방안을 논의했다. 오늘 회의의 기준은 새 기능을 크게 늘리는 것이 아니라 현재 업로드, 추출, 저장 흐름에서 사용자가 관찰할 수 없는 부분을 줄이는 것이다.

## 현재 흐름 검토

파일은 업로드 직후 documents 행으로 등록되고, 텍스트 추출이 끝나면 LLM 추출과 ingest가 이어진다. FastAPI 경로는 백그라운드 작업이므로 사용자는 상태 조회를 통해 진행 여부를 봐야 하고, Streamlit 경로는 동기 처리이므로 화면 안에서 바로 진행률을 보여줘야 한다. 두 경로 모두 추출 단계가 가장 오래 걸린다는 데 의견이 모였다.

## 문서 유형별 관찰 1

회의록은 문서 맨 위에 날짜와 참석자 목록이 있고, 뒤쪽 본문에서는 "앞서 말한 담당자가 진행한다"처럼 상단 정보를 참조하는 표현이 많다. 짧은 청크로 나뉘면 뒤쪽 청크는 날짜와 참석자를 직접 보지 못해 owner나 date가 비는 경우가 있다. 특히 회의록은 액션 아이템이 마지막 정리 섹션에 몰려 있어 청크 경계 영향을 많이 받는다.

## 문서 유형별 관찰 2

기획서는 중간에 표, 목록, 긴 설명 문단이 섞인다. 긴 문단을 글자 수로만 자르면 표의 행이 끊기거나, 결정의 배경과 결론이 서로 다른 청크로 분리된다. 이번 개선에서는 문단 경계를 우선하되, 정말 긴 문단만 글자 수 기준으로 자르는 절충안이 가장 작고 안전하다고 봤다.

## 분석 메모 A

업로드 페이지에서는 사용자가 파일 여러 개를 올릴 수 있다. 기존 파일 단위 진행바는 전체 파일 수 기준으로 어느 파일을 처리하는지 알려준다. 하지만 파일 하나 내부에서 LLM 추출이 몇 청크까지 끝났는지는 알려주지 못한다. 따라서 파일 단위 진행바는 유지하고, 파일 하나를 처리하는 함수 내부에 LLM 청크 진행바를 따로 두는 쪽이 화면 변화가 작다.

## 분석 메모 B

FastAPI API에서는 웹소켓이나 별도 작업 큐를 도입하지 않는다. 이미 status 엔드포인트가 있으므로 documents 테이블에 현재 청크와 전체 청크 숫자만 저장하면 된다. 값이 없을 때는 null로 내려주면 기존 클라이언트가 깨지지 않는다. 실패와 성공이 끝난 뒤에는 다시 null로 비워 진행 중 상태와 완료 상태를 구분한다.

## 분석 메모 C

청크 크기를 크게 늘리면 LLM 호출 수가 줄어든다. 호출 수가 줄면 비용과 대기 시간이 줄고, 문서 상단의 맥락이 뒤쪽 액션 아이템까지 유지될 가능성이 높다. 다만 매우 긴 문단은 그대로 보내면 모델 입력 한계를 넘을 수 있으므로 기존 슬라이스 폴백은 남겨야 한다. 이 폴백은 실제 긴 문단에서만 동작한다.

## 분석 메모 D

평가 스크립트는 데이터베이스에 쓰지 않는 방식으로 작성한다. 목적은 새 청크 분할이 실제 추출 품질을 얼마나 바꾸는지 보는 것이지, 저장 파이프라인 전체를 다시 검증하는 것이 아니다. 그래서 extract만 호출하고, 결과를 메모리에서 골든셋과 비교한다. 비용이 드는 LLM 호출이므로 pytest 자동 수집 대상도 아니어야 한다.

## 분석 메모 E

중복 추출은 청크 경계에서 자주 발생한다. 예전 방식은 경계마다 200자 오버랩이 들어가므로 같은 액션이나 리스크가 두 청크에서 모두 추출될 수 있다. 문단 경계로 나뉘는 새 방식은 문단 사이에서는 오버랩을 두지 않으므로 중복이 줄어들 가능성이 있다. 단일 초대형 문단의 폴백에서는 기존과 같은 오버랩을 유지한다.

## 분석 메모 F

데모 시나리오에서는 상태 표시가 중요하다. 사용자가 "멈춘 것 같다"고 느끼는 순간은 API가 실패했을 때보다 아무 변화 없이 기다릴 때다. 따라서 정확한 초 단위 예측보다 현재 완료된 청크 수와 전체 청크 수를 보여주는 것이 더 낫다. 이 정보는 서버와 로컬 Streamlit 모두 같은 콜백에서 받을 수 있다.

## 분석 메모 G

이번 작업에서는 시스템 프롬프트를 바꾸지 않는다. 프롬프트를 바꾸면 청크 크기 변경 효과와 지시문 변경 효과가 섞여 평가가 어려워진다. dedup 로직도 그대로 두어 청크 전략 차이만 비교한다. 평가 결과가 부족하면 이후에 별도 작업으로 프롬프트나 후처리를 조정한다.

## 분석 메모 H

회의 중반에는 실제 사용자 시나리오를 다시 검토했다. 사용자는 파일을 올린 뒤 아무 변화가 없으면 프로그램이 멈췄다고 판단한다. 특히 PDF나 긴 기획서는 텍스트 추출은 빠르게 끝나도 LLM 추출이 오래 걸린다. 따라서 화면에는 내부 단계명을 길게 설명하기보다 현재 청크 번호를 짧게 보여주는 것이 낫다.

## 분석 메모 I

서버 상태 조회는 폴링 방식으로 충분하다는 의견이 많았다. 별도 이벤트 스트림은 매력적이지만 지금은 클라이언트와 서버 양쪽 계약을 새로 만들어야 한다. 이미 documents 상태를 가져오는 엔드포인트가 있으므로 그 응답에 두 숫자를 더하는 것이 가장 작은 변경이다. 값이 null이면 진행 중이 아니라고 보면 된다.

## 분석 메모 J

문단 경계 분할은 회의록 작성 습관과도 맞다. 회의록은 안건, 결정, 액션 아이템처럼 빈 줄로 구분되는 구조가 많다. 이 구분을 보존하면 한 문단 안의 문맥이 잘리지 않는다. 문장 단위 파서는 언어별 예외가 많고 새 의존성을 요구할 수 있어 이번 범위에서는 제외한다.

## 분석 메모 K

평가용 문서는 실제 운영 문서처럼 완벽하지 않아도 된다. 중요한 것은 신구 청크 전략이 다르게 반응할 수 있는 구조다. 짧은 문서는 회귀 확인용이고, 중간 문서는 상단 메타데이터와 후반 액션의 거리를 벌린다. 긴 문서는 앞부분 논의와 뒷부분 결론을 떨어뜨려 중복과 누락을 확인한다.

## 분석 메모 L

진행 콜백은 추출 함수의 내부 구현을 외부에 노출하지 않는다. 외부는 done과 total만 받는다. 청크 분할 방식이 나중에 바뀌어도 콜백 계약은 그대로 유지된다. 콜백에서 예외가 나도 추출이 계속되어야 하는 이유는 UI 갱신 실패가 데이터 추출 실패로 이어지면 안 되기 때문이다.

## 분석 메모 M

부분 실패 처리는 기존 정책을 그대로 따른다. 일부 청크가 실패해도 성공한 항목은 저장할 수 있다. 진행률 콜백은 성공과 실패 모두에서 완료된 청크 수를 올린다. 사용자는 마지막 성공 메시지나 부분 실패 경고를 통해 결과를 확인한다. 이 정책을 바꾸면 저장 결과의 의미가 달라지므로 이번 범위가 아니다.

## 분석 메모 N

데이터베이스 컬럼은 정수 두 개면 충분하다. percent 값을 저장하면 반올림과 표시 기준을 서버가 결정해야 한다. done과 total을 저장하면 클라이언트가 필요한 방식으로 표시할 수 있다. total이 0인 상태는 만들지 않으며, 진행이 끝난 문서는 null로 되돌린다.

## 분석 메모 O

이 문서의 후반부 액션 아이템은 의도적으로 앞부분 참석자 목록과 멀리 떨어져 있다. 구버전 청크 크기에서는 뒤쪽 청크가 문서 날짜와 참석자 맥락을 잃을 수 있다. 새 청크 크기에서는 같은 문서 안에서 더 많은 상단 정보를 유지한다. 평가에서는 김민수와 박지현 액션의 owner와 date 정확도를 확인한다.

## 분석 메모 P

정확도 비교는 절대 점수보다 방향성이 중요하다. 같은 모델, 같은 문서, 같은 골든셋으로 legacy와 new를 비교하면 청크 전략이 만드는 차이를 볼 수 있다. 실행 횟수를 늘리면 모델 응답 흔들림을 평균낼 수 있지만 비용도 늘어난다. 기본값은 세 번으로 두고 수동 실행자가 조정한다.

## 분석 메모 Q

Streamlit의 파일 단위 진행바는 여러 파일 중 어느 파일을 처리하는지 알려주는 역할이다. 이 진행바를 제거하면 멀티파일 업로드 경험이 나빠진다. 따라서 파일 단위 진행바와 LLM 청크 진행바는 서로 다른 층에 둔다. 사용자는 먼저 전체 파일 순서를 보고, 각 파일 처리 중에는 청크 상태를 본다.

## 분석 메모 R

백엔드 API는 업로드 요청을 즉시 반환한다. 사용자는 doc_id를 받은 뒤 상태 엔드포인트를 호출한다. 이 구조에서는 progress_done과 progress_total이 서버에 저장되어야 한다. 메모리 변수로만 두면 프로세스 재시작이나 다른 요청에서 값을 볼 수 없다. documents 행에 두는 것이 현재 구조와 맞다.

## 분석 메모 S

이번 변경은 파일 크기 제한과 무관하다. 큰 파일을 더 많이 받는 기능이 아니라, 이미 받은 텍스트를 모델에 더 적절한 크기로 나누는 기능이다. 따라서 최대 파일 바이트 수는 바꾸지 않는다. 파일 제한을 바꾸려면 업로드 검증, 사용자 안내, 서버 메모리 사용량을 별도로 검토해야 한다.

## 분석 메모 T

테스트는 LLM을 호출하지 않는 선에서 끝낸다. 청크 분할은 문자열만으로 검증할 수 있고, 진행 콜백은 가짜 LLM 클라이언트로 확인할 수 있다. 평가 스크립트의 매칭 함수도 가짜 추출 항목으로 충분히 테스트할 수 있다. 실제 모델 호출은 수동 평가 명령에만 남긴다.

## 분석 메모 U

상태 응답의 새 필드는 기존 클라이언트가 모르면 무시할 수 있는 형태다. 필수 필드를 바꾸거나 중첩 구조를 새로 만들지 않는다. progress_done과 progress_total을 top-level에 두면 데스크톱이나 Streamlit 외부 클라이언트도 쉽게 사용할 수 있다. null은 진행 정보 없음이라는 뜻이다.

## 분석 메모 V

콜백은 각 청크 처리 직후 호출된다. 실패한 청크도 done 숫자는 증가한다. 사용자가 보는 진행률은 "성공한 청크"가 아니라 "처리가 끝난 청크" 기준이다. 이 차이를 지키지 않으면 실패한 청크에서 진행바가 멈춘 것처럼 보인다. 부분 실패 경고는 별도로 보여준다.

## 분석 메모 W

문단 경계 분할에서는 문단 사이 오버랩을 두지 않는다. 같은 문단이 두 번 들어가지 않으므로 중복 추출 가능성이 줄어든다. 다만 단일 문단이 너무 길면 기존 오버랩 방식으로 나누어 경계 손실을 줄인다. 이 예외는 긴 로그나 표를 위한 안전장치다.

## 분석 메모 X

평가 문서의 골든셋은 완전한 정답 데이터베이스가 아니다. 비교하려는 대표 항목만 담는다. 너무 많은 항목을 넣으면 리포트가 읽기 어려워지고, 너무 적으면 차이를 보기 어렵다. 이번 문서는 chunking 변화가 잘 드러나는 담당자, 날짜, 중복 항목 위주로 골든셋을 구성한다.

## 분석 메모 Y

업로드 처리 종료 시점에는 진행률을 반드시 지운다. 성공, 추출 실패, 저장 실패가 모두 해당된다. 진행률이 남아 있으면 status가 failed인데 progress가 2/3처럼 표시될 수 있다. 사용자는 실패한 작업이 아직 도는지 오해할 수 있다. 그래서 status 갱신 함수에서 함께 null로 되돌린다.

## 분석 메모 Z

이번 작업의 목표는 정확도와 관찰 가능성을 조금 높이는 것이다. 새 아키텍처를 만들지 않고, 기존 extract와 ingest 흐름에 필요한 훅만 넣는다. 이 방향이면 되돌리기도 쉽고, 문제가 생겼을 때 어느 부분이 원인인지 좁히기 쉽다. 후반 액션은 이 기준을 문서화하는 일이다.

## 분석 메모 AA

마지막 확인으로, 이 문서는 중간 길이 문서에서 구버전이 여러 청크로 나뉘고 새 버전은 한 청크에 머무르는 상황을 만들기 위한 fixture다. 본문 추가 문단은 골든셋에 직접 들어가지 않는다.

## 후반 정리

회의 초반에 정한 기준에 따라 구현 범위는 작게 유지한다. 컬럼은 progress_done과 progress_total 두 개만 추가한다. 상태 응답에는 두 값을 그대로 내려준다. Streamlit에서는 추출 중 문구를 "LLM 추출 중... (청크 done/total)" 형태로 갱신하고, 저장 단계에서는 "저장 중..."으로 바꾼다.

## 액션 아이템

김민수는 다음 주까지 API 문서에 progress_done과 progress_total 필드를 설명하고, 상태 응답 예시를 추가한다. 이 작업은 오늘 참석자 목록의 김민수가 담당한다.

박지현은 Streamlit 업로드 화면에서 추출 진행바가 완료와 실패 시 모두 사라지는지 확인한다.

## 결정

청크 기본 크기는 15000자로 늘리고, 문단 경계 기준으로 먼저 묶는 방식으로 확정했다. 단일 문단이 이 크기를 넘을 때만 기존 글자 수 슬라이스와 200자 오버랩을 사용한다.
