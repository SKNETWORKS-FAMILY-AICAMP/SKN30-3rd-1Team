## 주요 원인 분석

### 1. Legacy의 정규식 라우팅이 ‘질문 의미’보다 단어 표면을 우선했다

‘몇 %’, ‘누가’, ‘총 얼마’, ‘문제’, ‘수정했는가’, ‘처음/최종’ 같은 표현을 목록·상태 필터 신호로 받아들였다. 하지만 실제 질문은 단일 사실, 문서 부재, 시간 흐름, 원인 비교를 묻고 있었다. 그 결과 질문에 맞는 문단을 검색하지 않고 26건·55건·103건을 그대로 출력하거나 0건을 반환했다.

### 2. 구조화 행과 원문 문맥의 정보 밀도가 달랐다

64% 리텐션, 8초→1초, 출시일 유지 이유처럼 회의 본문에 들어 있는 세부 수치는 구조화된 decision/action/issue 행만으로 충분히 답하기 어렵다. Legacy는 잘못 라우팅된 뒤 원문 의미 검색으로 되돌아오지 않았다.

### 3. `filter_lookup`은 정답 후보를 좁히지 못하면 ‘우다다’ 출력했다

필터가 비거나 너무 넓으면 103건 전체를 보여줬다. 일부 문항은 정답 행이 목록 안에 있었지만, 사용자가 원하는 한 문장으로 종합하지 않았기 때문에 PARTIAL에 그쳤다. 출력 길이 p95가 7,360자인 이유도 이 경로다.

### 4. Agentic은 라우팅을 개선했지만 최종 해석 검증은 아직 필요하다

B4와 C1은 검색 실패가 아니다. 관련 근거가 들어왔는데도 ‘후속 버전 편입’을 ‘기존 결정 번복’으로, ‘Apple OAuth 처리’를 ‘Apple 앱 SDK 연동’으로 바꿔 읽었다. 따라서 도구 선택만 고치면 끝나는 문제가 아니고, 최종 답변 전에 질문의 범위(버전·시점·클라이언트/서버)를 근거와 다시 대조하는 검증이 필요하다.

## 이 보고서만으로 확정할 수 있는 것과 없는 것

- **확정 가능:** 이 스냅샷의 Modu 30문항에서는 Agentic이 Legacy보다 엄격 PASS가 9개 많고, 전체 목록 덤프를 없앴다.
- **확정 가능:** Legacy 오답/부분답 11개는 모두 `filter_lookup` 라우팅과 일치한다.
- **확정 가능:** Agentic의 두 실패는 도구 미호출이 아니라 최종 관계·역할 해석 오류다.
- **아직 미검증:** 이 골든셋은 30개 모두 의미 검색이 기대 경로이므로, 진짜 목록·집계 질문에서 structured tool이 Legacy보다 좋은지는 별도 평가가 필요하다.
- **아직 미검증:** 동일 질문을 여러 번 실행했을 때 판정이 안정적인지는 단일 스냅샷만으로 알 수 없다.
