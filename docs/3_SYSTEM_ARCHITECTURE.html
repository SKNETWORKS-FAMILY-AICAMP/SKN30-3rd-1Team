PaiM 시스템 아키텍처

대상 브랜치: main  ·  갱신: 2026-07-04  ·  코드베이스 실측 기준 (README와 다르면 이 문서 우선)

1. 개요

PaiM은 회의록·문서와 GitHub 저장소 활동을 하나의 "살아있는 프로젝트 메모리"로 통합하는 LLM 기반 AI 프로젝트 매니저입니다.

  • 입력: 회의록(.md/.txt/.pdf) 업로드, GitHub repo 연결
  • 처리: LLM이 결정(decision)·액션(action)·이슈(issue)·리스크(risk)로 구조화 추출 → MySQL + ChromaDB 이중 저장
  • 관찰: repo sync 시 머지된 PR과 열린 액션을 LangGraph 기반 Reconciler가 대조해 완료 제안 생성 (승인은 항상 사람)
  • 질의: 질문 의도(조회/조망/탐색)에 따라 SQL 직조회 / 요약 직접 응답 / 하이브리드 RAG로 라우팅
  • UI: Tauri + React 데스크톱 앱 (macOS/Windows). Streamlit 프로토타입 UI는 레거시로 유지

3. 백엔드 아키텍처

3.1 요청 진입 및 부트스트랩 backend/main.py

  • FastAPI 앱 생성 후 api/*, chat/router, github/router를 /api/v1 prefix로 등록 (github/router는 자체 /github/app prefix 사용)
  • CORS는 데스크톱 앱의 Tauri origin만 허용
  • lifespan에서 기동 시: stale 문서/repo 작업 복구, 레거시 프로젝트 멤버십 백필, ChromaDB 벡터 백필 실행 + stale_watchdog 백그라운드 태스크 기동
  • 백엔드는 127.0.0.1에만 바인딩 — 로컬 우선, LAN 미노출

3.2 API 레이어

모듈라우트 예시역할

upload.py/projects/{id}/documents, .../memory문서 업로드, 메모리 CRUD

auth.py—개발용 임시 인증(DEV_USER_ID)

3.3 pipeline — 기억 만들기

문서/repo 콘텐츠 → extractor.extract() │ 소스 타입별 지침 분기: │ 회의록 → 결정/액션(담당자)/이슈/리스크 │ README → 로드맵/TODO만 액션, 설치 안내문 제외 │ 커밋 → 완료 상태 액션 또는 결정 │ 열린 이슈/PR → 현재 문제·진행 중 작업 │ (대용량 문서는 문단 경계 청크 분할 후 병합·중복 제거) ▼ ingestor.ingest() │ MySQL(memory 테이블, 구조화) 저장 └→ ChromaDB(벡터, 의미 검색) 저장

3.4 reconciler — 기억이 스스로 갱신

pr_actions.py가 LangGraph 그래프(build_reconciler_graph)로 머지된 PR과 열린 액션을 배치 대조합니다.

  • 저장소별 last_reconciled_pr 워터마크로 증분 처리(이미 본 PR 재검사 방지)
  • LLM 매칭은 high/medium 확신 + 한 줄 근거가 있을 때만 제안 생성 — 애매하면 보고하지 않음
  • 제안은 memory_suggestions에 삽입, 사람이 accept/reject로 확정할 때만 memory.completed_at 갱신

3.5 retriever — 기억에 묻기 (의도 라우터)

경로판단처리
조회형classifier.py 키워드 규칙 → mysqlmysql_search.py: 필터 추출 → SQL 직조회 → 결정론 템플릿 (정답 보장)
조망형전체 요약 요청project_memory 응축 요약을 검색 없이 직접 컨텍스트로 사용
탐색형chroma / bothqa_engine.py: 멀티쿼리 재표현 → BM25(한국어 형태소)+dense RRF 융합 → 출처 있는 LLM 답변

memory_vector.py는 memory 행 생성/수정 시 ChromaDB에도 보조 인덱싱해 탐색형 검색이 구조화 데이터도 커버하도록 합니다.

3.6 graph.py — LangGraph 오케스트레이션

State(TypedDict)를 노드 간 계약으로 고정하고, 노드는 pipeline/retriever 로직을 재사용하는 얇은 래퍼입니다.

  • 적재 그래프: 문서 → [저장] → [메모리] → END
  • 질의 그래프: 질문 → [섹션(stub)] → [Q&A] → [검증] → (부족 시 재검색, MAX_RETRY=1) → [계획 제안] → [검증] → (부족 시 재기획) → [응답] → END — 계획 생성 실패해도 답변은 유지(best-effort)

3.7 chat — 암호화 세션 대화

  • session_store.py: 세션·메시지를 security/session_crypto.py(AES-256-GCM, SESSION_MEMORY_KEY)로 암호화 저장
  • context_builder.py: tiktoken으로 토큰 수 계산 후 시스템 프롬프트+요약+최근 메시지를 토큰 예산 안에서 조립

3.8 llm — 프로바이더 추상화

용도가 다른 두 팩토리가 존재합니다.

  • factory.py (get_llm_client): 구조화 추출 전용, SDK 직접 래핑 (Claude/OpenAI/Google Client)
  • chat_model_factory.py (get_chat_model): 자유 대화형 Q&A/RAG용 LangChain BaseChatModel 반환, LLM_PROVIDER 환경변수로 openai/claude/google/local 선택, tier="fast"|"quality" 지원
Google Gemini는 중첩 스키마 구조화 출력을 지원하지 않아 구조화 추출에는 사용하지 않고 Q&A에만 사용합니다.

3.9 github — GitHub App 연동

github/router.py가 설치(install) 세션 발급, OAuth 콜백, JWT 서명, repo preview를 처리합니다.

설치 세션은 현재 인메모리(_sessions dict)로 관리 — 코드 주석상 다중 워커/정식 사용자 인증 도입 시 DB/Redis로 이전 예정입니다.

3.10 db — 저장소

  • mysql.py: PyMySQL 커넥션 헬퍼
  • chroma.py: OpenAI 임베딩 전용 별도 컬렉션(cosine space) — 기본 chromadb 임베딩(384-dim, L2)과 분리
  • schema.sql: users, projects, project_members, documents, repositories, memory, memory_sources, memory_suggestions, chat_sessions, chat_messages, chat_summaries, project_memory (12개 테이블)
  • migrate_v2~v5.sql: 문서 처리 상태 → 액션 완료/정렬 → 마감일 → PR 워터마크+제안 인박스 순 idempotent 마이그레이션 (컨테이너 기동 시 자동 실행)

4. 데스크톱 아키텍처 공식 UI

Tauri 2(Rust 셸) 위에 React 19 + TypeScript로 구성됩니다.

  • App.tsx가 최상위 앱 셸(약 5,800줄)로 프로젝트 선택·채팅·레이아웃을 오케스트레이션
  • ProjectMemoryPanel.tsx(메모리/브리핑), GithubPanel.tsx(repo 연동/제안 인박스), projectFiles.tsx(문서 업로드)가 주요 기능 패널
  • paimApi.ts/github.ts가 백엔드 REST 호출 캡슐화, settings.ts가 로컬 설정 관리
  • src-tauri/는 네이티브 윈도우/트레이/권한을 구성하는 Rust 셸 — 비즈니스 로직 없음
  • CI(.github/workflows/release.yml)가 태그 push 시 macOS/Windows용으로 빌드해 릴리즈

5. 레거시: frontend/ (Streamlit) Legacy

frontend/는 Streamlit 기반 초기 프로토타입 UI입니다. README·로드맵 어디에도 언급되지 않으며 공식 사용자 UI는 desktop/(Tauri)로 대체되었습니다.

pyproject.toml의 tool.hatch.build.targets.wheel.packages에는 여전히 포함되어 패키징되고, tests/test_frontend_contract.py가 meeting_notes/ fixture로 계약 테스트를 유지하고 있어 완전히 죽은 코드는 아니지만, 신규 기능 개발은 desktop/ 기준으로 이루어집니다.

8. 핵심 흐름

8.1 문서 업로드 → 기억 적재

사용자 업로드 (api/upload.py) → storage.py 로 파일 저장 (BackgroundTasks로 비동기 처리) → pipeline/extractor.py: 소스 지침 기반 LLM 구조화 추출 → pipeline/ingestor.py: MySQL(memory) + ChromaDB 이중 저장 → 문서 상태(status)를 polling(GET .../documents/{id}/status)으로 확인

8.2 GitHub repo 동기화 → 완료 제안

repo 연결/동기화 (api/repository.py: POST .../sync) → GitHub App API로 머지 PR·열린 이슈/README/커밋 수집 → pipeline/extractor.py 로 README/커밋/이슈 구조화 적재 (소스 타입별 지침) → reconciler/pr_actions.py: 워터마크 이후 머지 PR × 열린 액션 LLM 배치 대조 → high/medium 확신 매칭만 memory_suggestions 에 제안 생성 → 사용자가 GithubPanel.tsx 의 제안 인박스에서 승인/거절 (api/suggestion.py) → 승인 시에만 memory.completed_at 갱신

8.3 질문 → 답변

사용자 질문 (api/query.py 또는 chat/router.py 세션 질의) → retriever/query_intent.py: 의도 분류 ├─ 조회형 → mysql_search.py → SQL 직조회 → 결정론 템플릿 ├─ 조망형 → project_memory 응축 요약 직접 사용 └─ 탐색형 → qa_engine.py → 멀티쿼리 + BM25/dense RRF → LLM 생성 → graph.py 검증 루프 → (세션 질의) context_builder.py 로 암호화 대화 이력과 함께 컨텍스트 조립 → 답변 + 출처 반환

8.4 델타 브리핑

앱 재오픈 (api/delta.py: GET/POST .../delta, .../briefing/delta) → 마지막 확인 시점 이후 변경분 조회 (완료된 액션·새 완료 제안 → 새 결정/액션/이슈/리스크 → 마감 임박/기한 초과 순) → LLM이 스탠드업 대체 브리핑으로 요약 (약 8문장)

9. 설계 원칙

  • 정확도 > 재현율 — Reconciler의 완료 매칭은 애매하면 보고하지 않음. 놓친 제안은 다음 동기화나 사람이 잡을 수 있지만, 틀린 완료 처리는 신뢰를 무너뜨림.
  • 파괴적 변경은 제안-승인, 추가는 자동 — 메모리 적재는 자동, 완료 처리처럼 상태를 바꾸는 일은 반드시 사람의 승인을 거침 (human-in-the-loop).
  • 자기검증하는 답변 그래프 — 탐색형 Q&A는 검색 → 답변 → 검증 → (부족 시) 질의 확장 후 재검색 → 계획 제안까지 도는 LangGraph. 계획 생성 실패해도 답변은 유지(best-effort).
  • 로컬 우선 + 암호화 — 백엔드는 127.0.0.1에만 바인딩, 세션 대화는 AES-256-GCM 암호화 저장.

10. CI/CD

.github/workflows/release.yml이 버전 태그(v*) push 시 desktop/(Tauri) 앱을 macOS .dmg, Windows -setup.exe/.msi로 빌드해 GitHub Releases에 게시합니다. 백엔드는 별도 CI 없이 로컬(docker compose + uv run uvicorn) 또는 start-paim.bat(Windows 원클릭)으로 구동합니다.
