# 출처 인용(citation) 기능 — 프론트엔드 핸드오버

- 작성: 2026-07-22 (TASK-007 반영, origin/main `5240f40` 기준)
- 관련 API: `POST /api/v1/projects/{id}/query` — [API_명세서.md](API_명세서.md)
- 백엔드 변경은 완료·병합됨. 이 문서는 프론트가 알아야 할 계약 영향과 선택적
  UI 작업을 정리한다.

## 1. 무엇이 바뀌었나

질의 답변(`answer`)의 근거에 **실제 출처 파일명**이 붙기 시작했다. 이전에는
답변이 "구조화 기록에 따르면…"처럼 유형 이름만 언급했으나, 이제는 근거의
실제 파일명을 `(출처: 파일명)` 형태로 인용한다.

- **`answer` 본문**: `(출처: 2026-03-02_킥오프.md)` 같은 마커가 문장/불릿에
  포함된다. 여러 출처는 `(출처: a.md, b.md)`로 묶이거나 각각 표기될 수 있다.
- **저장소 연결 파일**: 동명 충돌 방지를 위해 `(출처: README.md (repo#7))`
  처럼 `repo#N` 접미가 붙을 수 있다.
- **`sources` 배열**: **변경 없음.** 답변 근거로 검색된 출처 파일명 리스트가
  기존 그대로 제공된다. `answer` 마커와 별개다.
- **`debug.mysql_rows[]` / `debug.chroma_chunks[]`**: `source_label` 필드가
  추가됐다(additive). `answer` 마커가 이 라벨을 사용한다. 디버그 표시에만
  쓰므로 무시해도 된다.

**요청 스키마·엔드포인트·상태 코드·`sources` 계약은 그대로**다. 즉 파싱을
안 바꿔도 앱은 동작하며, 아래 UI 작업은 전부 **선택**이다.

## 2. 지금 데스크톱이 어떻게 동작하나 (수정 전)

- `answer`를 그대로(마크다운/텍스트) 렌더링한다면, 이제 답변 안에 `(출처: …)`
  문구가 **그대로 노출**된다 — 깨지지는 않지만 다소 장황해 보일 수 있다.
- `sources` 배열을 이미 별도 목록/칩으로 보여주고 있다면, 그 동작은 불변이다.

## 3. 필요한 UI 작업 (선택)

### 3-1. 출처 마커를 칩/링크로 승격 (권장)
`answer`에서 `(출처: …)` 마커를 정규식으로 추출해 인라인 텍스트 대신 출처
칩·툴팁·문서 링크로 렌더링하면 가독성이 좋아진다.

- 추출 패턴 예: `\(출처:\s*([^)]*)\)` — 단, 파일명에 `)`가 있는 경우
  (예: `회의록) 최종.md`, `repo#N` 접미)가 있어 **완벽 파싱은 어렵다.**
- **정확 매칭(exact matching)은 `debug.mysql_rows[].source_label` /
  `debug.chroma_chunks[].source_label`의 라벨 집합과 대조**하라. 답변 마커가
  실제로 이 `source_label` 값을 쓰므로(예: `README.md (repo#7)`) 마커 문자열과
  정확히 일치한다. 백엔드 채점기도 이 라벨 집합으로 대조한다.
- **`sources` 배열은 표시용 raw 목록**이다 — 검색된 출처 파일명의 사람이 읽는
  리스트이며, 저장소 파일의 `repo#N` 같은 충돌 방지 접미가 없어 마커와 1:1로
  일치하지 않을 수 있다. 정확 대조 용도로 쓰지 말고 요약 표시에만 쓰라.
- **첨부(attachment) 질의는 예외**: 첨부 전용 질의의 답변 마커
  `(출처: 첨부파일명)`은 `debug.mysql_rows`/`chroma_chunks`의 `source_label`이
  아니라 **`debug.attachments`(첨부 파일명 배열)와 `sources`** 에만 등장한다.
  따라서 첨부 route(`route: "semantic"`, `debug.router_stage: "attachment"`)에서는
  `debug.attachments` ∪ `source_label` 집합으로 대조하라.
- 칩 클릭 시 원문 문서로 이동하려면 파일명↔문서 매핑이 필요하다 —
  문서 목록 API(`GET .../documents`)의 `filename`과 매칭한다.

### 3-2. 마커를 그대로 두기 (최소 작업)
아무 작업도 안 하면 `(출처: 파일명)`이 답변 본문에 텍스트로 남는다. 기능상
문제는 없다. 다만 사용자에게 파일명이 노출되므로, 내부 파일명이 민감하면
3-1의 칩 처리를 권한다.

## 4. 주의점

- **적용 범위(응답 `route`별로 다름)**: 인라인 `(출처:)` 마커는 `route`가
  `semantic`(RAG 검색)이거나 첨부(`attachments`) 질의일 때 **best-effort로만**
  붙는다. `route: "filter_lookup"`(구조화 조회 템플릿)·`route: "overview"`
  (프로젝트 조망 요약) 응답에는 **마커가 없을 수 있다** — 이 경로들은 답변을
  템플릿/요약으로 구성하기 때문이다. 따라서 마커 부재를 오류로 처리하지 말고,
  `route`를 확인해 마커가 없으면 `sources`(표시용)로 폴백하라.
- **마커 파싱은 관대하게**: 답변 생성은 LLM이라 마커 형식이 100% 균일하지
  않을 수 있다(예: `(출처: 파일명 원문)`처럼 접미가 붙는 경우 관측됨).
  파싱 실패 시 마커를 그대로 두는 폴백을 두라.
- **`repo#N` 접미**: 저장소 파일에서만 나타난다. 문서 업로드만 쓰는 프로젝트
  에서는 순수 파일명만 나온다.
- **프로젝트 메모리 요약 근거**: 답변이 프로젝트 전반 요약에 기댄 경우엔 파일
  출처가 없을 수 있다(요약은 합성물). 이때는 `sources`가 근거 파일을 보완한다.

## 5. 백엔드에 요청 가능한 것 (지금이 변경 적기)

- 마커 대신 **구조화된 인용 배열**(예: `answer_citations: [{label, span}]`)이
  프론트에 더 편하면 백엔드에 요청 가능하다. 현재는 `answer` 인라인 마커 +
  `sources` 배열 + `debug.source_label` 조합으로 제공한다.
- 파일명↔문서 ID 매핑이 필요하면 응답에 doc_id 포함을 요청할 수 있다.
