# PaiM Overview 구조 개선 검토 보고서

> 상태: 핵심 계약 구현 완료 · 데이터 검증 통과 · 합성 한계 확인
> 작성일: 2026-07-22
> 기준 평가: `paim-route-balanced-golden-20260722` 40문항

## 1. 결론

PaiM의 overview는 **현재 프로젝트 요약본과 활성 Action Plan 전체를 LLM에 제공하는 단순한 도구**로 정의하는 것이 적절하다.

현재처럼 액션을 상태별로 일부만 잘라 전달하면 컨텍스트 비용은 줄지만, 중요한 액션이 LLM 입력 전에 사라진다. 실제 평가 코퍼스는 Modu 55개, CS-Bot 30개 액션이므로 전체 전달이 가능한 규모다. 지금 단계에서는 임의의 Top-N 샘플링보다 정보 보존을 우선해야 한다.

권장 계약은 다음과 같다.

```text
프로젝트의 현재 요약
        +
활성 Action Plan 전체(상태·담당자·기한·출처 포함)
        ↓
LLM이 질문에 필요한 항목만 선택해 답변
```

이 변경은 특정 골든 질문이나 프로젝트 이름을 사용하지 않는다. 모든 프로젝트에 같은 데이터 계약을 적용하므로 과적합 위험이 낮다.

## 2. 평가에서 확인된 현상

overview가 필요한 문항은 순수 overview 8개와 mixed 4개를 합쳐 12개였다.

| 지표 | 결과 | 해석 |
|---|---:|---|
| overview 도구 선택 | 11/12 (91.7%) | 라우팅은 대부분 정상 |
| 최종 답변 PASS | 7/12 (58.3%) | 올바른 도구를 골라도 정보 누락 발생 |
| 최종 답변 PARTIAL | 4/12 (33.3%) | 핵심 일부는 맞지만 일정·수치·액션 누락 |
| 최종 답변 FAIL | 1/12 (8.3%) | 잘못된 도구와 인자 조합으로 사실 반전 |

따라서 주요 병목은 단순한 overview 라우팅 정확도가 아니다. **도구가 제공하는 데이터의 완전성과 최종 답변 합성**이 더 큰 문제다.

### 대표 사례

- `C-OV02`: `risk=0`이 payload에 없었고, 프로덕션 배포 액션이 최신 5개 밖으로 밀려 누락됐다.
- `M-OV04`: 6월 8일 1.1 일정은 요약에 있었지만 최종 답변에서 빠졌다.
- `C-MX01`: 검색 청크에 `55% → 87%, +32%p`가 모두 있었지만 87%만 답했다.
- `C-OV04`: overview 대신 category별 구조화 조회를 호출하고 issue에 action 전용 `completion_status=open`을 적용해, 이슈가 11건인데도 없다고 답했다.

## 3. 현재 구조의 문제

### 3.1 Action Plan이 아니라 일부 샘플만 제공한다

현재 overview 컨텍스트는 액션을 다음과 같이 제한한다.

- `open`: 최대 8개
- `completed`: 최대 5개
- `unknown`: 최대 5개

평가 스냅샷의 기존 액션은 대부분 `unknown`이므로 실제로는 최신 5개만 LLM에 전달된다. 이 선택 기준은 중요도가 아니라 저장 시각이다. 중요한 작업이 여섯 번째 이후에 있으면 모델은 그 작업을 볼 수 없다.

### 3.2 요약이 상세 사실을 대신하고 있다

decision·issue·risk는 개수만 전달되며 실제 대표 행은 전달되지 않는다. 상세 내용은 5문장짜리 `project_memory`에 의존한다. 이 요약은 기존 요약과 새 항목을 다시 압축하는 방식이어서 시간이 지날수록 세부 정보가 빠질 수 있다.

### 3.3 존재하지 않는 category의 0건이 사라진다

SQL `GROUP BY` 결과를 그대로 사용하므로 행이 없는 category는 응답에 포함되지 않는다. CS-Bot에서는 risk 행이 없어 `risk=0`이 아니라 risk 키 자체가 사라졌다. LLM은 없는 키를 0으로 안전하게 복원할 수 없다.

### 3.4 잘못된 필터 조합을 실행 단계에서 막지 않는다

`completion_status`는 action 전용이지만 다른 category에도 전달할 수 있다. 설명만으로 LLM의 잘못된 호출을 완전히 막을 수 없으며, 런타임 검증이 필요하다.

### 3.5 전체 데이터를 주는 것과 전부 출력하는 것을 혼동하고 있다

LLM 입력에 전체 액션을 제공해도 사용자 답변에 전체 액션을 나열할 필요는 없다. 데이터 제공 범위와 답변 범위는 별개의 계약이다.

- 도구: 필요한 정보를 잃지 않도록 전체 데이터를 제공한다.
- LLM: 사용자가 요청한 정보만 골라 간결하게 답한다.
- 사용자가 “전부 보여줘”라고 했을 때만 전체 목록을 출력한다.

## 4. 권장 Overview 데이터 계약

```json
{
  "overview_summary": "현재 유효한 프로젝트 기록을 기준으로 작성한 요약",
  "category_stats": {
    "decision": 24,
    "action": 55,
    "issue": 7,
    "risk": 0
  },
  "action_plan": {
    "total": 55,
    "status_counts": {
      "open": 8,
      "completed": 5,
      "unknown": 42
    },
    "items": [
      {
        "id": 96,
        "content": "5/18 안드로이드 출시 실행",
        "owner": "박현우",
        "date": "2026-05-15",
        "due_date": null,
        "completion_status": "unknown",
        "completion_status_source": null,
        "source": "2026-05-15_출시최종점검_1.1로드맵확정.md"
      }
    ]
  }
}
```

### 계약 원칙

1. `active_memory`의 action 행을 누락 없이 제공한다.
2. 임의의 `LIMIT 5/8`을 두지 않는다.
3. `open`, `completed`, `unknown`을 그대로 전달한다.
4. `unknown`을 open·진행 중·미완료로 바꾸지 않는다.
5. 담당자, 기한, 기록 날짜, 출처를 함께 제공한다.
6. 정렬은 표시 편의를 위한 것일 뿐 행을 제거하는 기준으로 사용하지 않는다.
7. LLM은 질문과 관련된 액션만 답하고, 전체 출력 요청이 있을 때만 모두 나열한다.

`category_stats`는 legacy 호환과 0건 표현을 위해 네 category를 항상 포함한다. `action_plan.status_counts`는 요약 문구와 액션 행이 충돌할 때 사용하는 결정적 상태 집계다. 정확한 조건별 개수가 필요하면 `query_structured_memory`를 함께 사용하며, overview 답변이 항상 통계를 낭독할 필요는 없다.

## 5. 과적합 방지 원칙

이번 변경에서 다음 방식은 사용하지 않는다.

- 평가 문항의 표현을 정규식으로 등록
- `C-OV04` 같은 특정 ID나 프로젝트명을 예외 처리
- 골든셋에 등장한 액션을 우선순위 목록으로 하드코딩
- 특정 담당자나 날짜를 prompt 예시로 추가
- 동일 골든셋 점수만 보고 prompt를 반복 조정

대신 다음 공통 계약만 적용한다.

- 프로젝트에 관계없이 같은 overview schema 사용
- 모든 활성 action 행 보존
- 상태 의미를 schema와 런타임에서 동일하게 검증
- 질문이 요구한 필드를 빠짐없이 답하도록 일반적인 출력 규칙 사용
- 기존 골든셋 외의 새로운 표현과 새로운 프로젝트로 별도 검증

## 6. 최소 변경 범위

### 필수

1. `_fetch_overview_context`에서 상태별 Top-N 쿼리를 action 전체 조회 한 번으로 교체한다.
2. `get_project_overview`가 `overview_summary + action_plan.items`를 반환한다.
3. 도구 설명에 “전체 데이터에서 필요한 항목만 선택하며, 요청 없이는 전부 나열하지 않는다”를 명시한다.
4. Agentic prompt에 `unknown` 상태를 추론하지 않는 규칙을 명시한다.

### 별도 안전장치

5. non-action category에 `completion_status`가 들어오면 실행을 거부한다.
6. 전체 현황·상태·브리핑 질문은 category 단어가 함께 있어도 overview를 우선한다.

### 이번 단계에서 하지 않을 것

- 중요도 점수 모델 개발
- Action Plan 요약용 별도 LLM 추가
- pagination 또는 새로운 저장소 도입
- 프로젝트별 threshold 설정
- 골든셋 전용 prompt 추가

액션이 실제로 컨텍스트 한도를 압박하는 규모가 된 뒤에만 pagination이나 단계적 요약을 검토한다.

## 7. 검증 계획

### 데이터 계약 테스트

- Modu 55개와 CS-Bot 30개 action이 모두 반환되는지 확인
- 첫 행과 마지막 행이 모두 포함되는지 확인
- 상태별 합계가 전체 action 수와 일치하는지 확인
- `unknown`이 그대로 보존되는지 확인
- source와 owner가 누락되지 않는지 확인

### 응답 행동 테스트

- 일반 브리핑에서는 전체 액션을 쏟아내지 않고 핵심만 선택
- 특정 담당자·주제 질문에서는 전체 데이터 중 관련 액션만 선택
- “Action Plan을 전부 보여줘”에서는 전체 항목 출력
- 완료 여부가 불명확한 액션을 진행 중이라고 단정하지 않음
- 요약과 액션 데이터가 충돌하면 구체적인 액션 데이터를 우선

### 과적합 검증

- 기존 12개 overview 관련 문항은 회귀 지표로만 사용
- 같은 의미의 새로운 표현을 별도 holdout으로 추가
- 기존 골든에 없는 새 프로젝트로 overview 평가
- 라우팅, 데이터 완전성, 답변 정확도를 각각 분리해 측정

## 8. 기대 효과와 남는 한계

### 기대 효과

- 중요한 액션이 샘플 상한 때문에 사라지는 문제 제거
- LLM이 질문별로 적절한 액션을 선택할 수 있는 근거 확보
- 프로젝트별·질문별 예외 없이 동일하게 동작
- `unknown` 상태에 대한 잘못된 추론 감소
- overview와 mixed-tool 답변의 누락 감소

### 남는 한계

- 요약 자체가 오래되거나 부정확하면 overview 방향이 흔들릴 수 있다.
- 전체 데이터를 받은 뒤에도 LLM이 일부 사실을 빠뜨릴 수 있다.
- C-OV04 유형의 잘못된 도구 선택은 별도 라우팅·인자 검증이 필요하다.
- 액션이 수백~수천 건으로 늘어나면 전체 전달 비용을 다시 측정해야 한다.

좋은 모델은 최종 합성 누락을 줄일 수 있지만, 데이터 계약의 빈틈을 대신할 수는 없다. 우선 전체 Action Plan을 보존하고, 그다음 동일 평가와 별도 holdout으로 모델 영향을 비교하는 순서가 적절하다.

## 9. 최종 권고

**채택 권고:** `get_project_overview = 현재 overview 요약 + 활성 Action Plan 전체`

이 안은 현재 규모에서 가장 단순하고, 정보 손실을 줄이며, 특정 평가 질문에 맞추지 않는 공통 해법이다. 우선 이 최소 계약만 적용한 뒤 동일 골든셋과 별도 holdout을 다시 실행하고, 실제 컨텍스트 비용이 문제가 될 때만 다음 최적화를 검토한다.

## 10. 구현 및 검증 결과

2026-07-22에 필수 범위 4개를 구현했다.

- `_fetch_overview_context`의 상태별 `LIMIT 8/5/5` 조회를 `active_memory` action 전체 조회 한 번으로 교체
- 반환 계약을 `overview_summary + action_plan.total/items`로 변경
- legacy 호환용 `category_stats`를 0건까지 채워 유지하고 `action_plan.status_counts`를 결정적으로 계산
- 일반 브리핑에서는 핵심만 선택하고, 명시적인 전체 요청에서만 전부 출력하도록 도구 설명과 prompt 보강
- `completion_status=unknown`을 open·미완료·진행 중으로 바꾸지 않도록 명시

검증 결과는 다음과 같다.

| 검증 | 결과 |
|---|---|
| Overview 관련 단위 테스트 | 21개 통과 |
| 전체 Python 회귀 테스트 | 363개 통과 |
| 개발 DB 데이터 계약 | action 8건 중 8건 반환, `unknown` 8건 보존 |
| 실제 LLM 일반 브리핑 | 도구 입력은 8건 전체, 출력은 3문장으로 선택 요약 |
| 실제 LLM 전체 목록 요청 | 8건 전체 출력, 담당자와 `unknown` 상태 유지 |
| 55건 혼합 상태 holdout | 도구는 55/55 반환, 상태 집계 `19/18/18` 정확 |
| 기본·상위 모델 상태 합성 | 오래된 요약의 “진행 중”과 `unknown`을 함께 말해 PARTIAL |
| 55건 전체 출력 요청 | 전체 데이터는 전달됐지만 기본 모델이 목록 재현에 실패 |

별도 중요도 모델, 추가 LLM, pagination, 프로젝트별 예외, 골든 문항 정규식은 추가하지 않았다. 따라서 이번 변경은 데이터 손실을 없애는 공통 계약에만 한정된다.

데이터 계층의 Top-N 누락은 해결됐다. 다만 현재 개발 프로젝트의 `overview_summary`에는 모든 액션이 `unknown`인데도 “작업이 진행 중”이라는 과거 생성 문장이 남아 있다. `gpt-4.1-mini`와 `gpt-4.1` 모두 이 문장과 정확한 상태 집계를 함께 답해 모순이 발생했다. 이는 모델 등급만 올려서 해결되지 않았으며, 다음 작업에서는 project_memory 생성·갱신 시점의 상태 정합성을 별도로 다뤄야 한다.

기존 Route-Balanced 40문항의 원본 패키지와 격리 평가 DB가 현재 로컬에 없어 동일 조건 전체 재평가는 실행하지 못했다. 기존 결과 파일은 보존되어 있지만 새 결과와의 정당한 전후 비교에는 원본 데이터셋과 동일 DB 재구성이 필요하다.
