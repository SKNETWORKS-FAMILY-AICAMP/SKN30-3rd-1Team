# CS-Bot Agentic Tool Routing 평가 보고서

> 이 문서는 `legacy_csbot.jsonl`과 `agentic_csbot.jsonl`에 저장된 **평가 당시 JSONL 스냅샷**을 대조한 보고서다. 이후 코드·프롬프트·데이터 변경은 이 결과에 반영되지 않는다.

## 결론 요약

동일한 CS-Bot 골든셋 30문항에서 legacy는 `PASS 15 / PARTIAL 6 / FAIL 9`, agentic은 `PASS 26 / PARTIAL 3 / FAIL 1`이었다. 완전 정답률은 **50.0%에서 86.7%로 36.7%p 상승**했다. PARTIAL을 0.5점으로 환산하면 **60.0%에서 91.7%**로 상승한다.

| 지표 | Legacy | Agentic | 차이 |
|---|---:|---:|---:|
| PASS | 15/30 (50.0%) | 26/30 (86.7%) | +11문항, +36.7%p |
| PARTIAL | 6/30 (20.0%) | 3/30 (10.0%) | -3문항 |
| FAIL | 9/30 (30.0%) | 1/30 (3.3%) | -8문항 |
| 평균 답변 길이 | 1,196.9자 | 245.9자 | -79.5% |
| 중앙 답변 길이 | 321.5자 | 247자 | -23.2% |
| 최대 답변 길이 | 4,780자 | 480자 | -90.0% |
| 평균 지연 | 6,430.8ms | 6,574.3ms | +2.2% |
| 중앙 지연 | 8,035.2ms | 6,210.9ms | -22.7% |
| HTTP 200 | 30/30 | 30/30 | 동일 |

Legacy 경로는 `semantic 16 / filter_lookup 9 / overview 5`였다. Agentic은 30문항 모두 `search_project_evidence` 도구를 한 번 사용했다. 이 골든셋의 기대 경로도 30문항 모두 `semantic`이므로, agentic이 전부 semantic 검색을 선택한 것은 이 평가에서는 맞는 행동이다. 다만 이 결과만으로 목록·개수 질문에서 구조화 도구 선택이 잘 되는지는 증명되지 않는다.

### 왜 결과가 이렇게 나왔는가

1. **Legacy 라우팅 오분류가 가장 큰 원인이다.** `overview`로 간 B1·B4·B5·B6은 프로젝트 요약만 보고 “기록에 없다”고 답했다. `filter_lookup`으로 간 B3·B9·B10·C1·C2·C3·C6·C8·C10은 질문에 답하지 않고 15~58건의 기록을 그대로 쏟아냈다.
2. **정규식/필터 방식은 질문의 관계를 이해하지 못했다.** “6/16과 6/27이 동일한가”, “전후 수치가 어떻게 변했는가” 같은 비교 질문을 단순 분류·날짜·완료 필터로 바꾸면서 핵심 관계가 사라졌다.
3. **Agentic 방식은 질문을 먼저 이해하고 근거 검색을 선택해 대부분 복구했다.** Legacy가 실패한 9문항 중 B1·B3·B4·B5·B9·B10·C2·C10을 직접 답변으로 바꾸었다. C3도 전체 목록 속 정답 행을 사람이 찾아야 했던 상태에서 바로 답하도록 개선됐다.
4. **Agentic의 남은 문제는 라우팅보다 검색·서술 품질이다.** B6은 정답 청크(1차 개선 후 1.9초)를 놓치고 이전 2.3초 수치와 `+0.6초`를 결합해 2.9초라고 답했다. B7은 `미정 → 구체화`를 설명하면서도 첫 문장에서 “일부 조정”이라 표현해 흐름을 흐렸다. B10은 전체 fallback 18%는 답했지만 반품·교환 31%를 누락했다. C5는 “최종 결정 없음” 대신 7/3에 확정될 것이라고 과도하게 단정했다.
5. **속도는 평균과 중앙값이 엇갈린다.** 평균 지연은 agentic이 2.2% 느렸지만 중앙값은 22.7% 빨랐다. 따라서 “전반적으로 더 빠르다”고 단정할 수는 없고, 긴 꼬리 지연을 별도로 확인해야 한다.

### 판정 기준과 원문 표시

- `PASS`: 골든의 핵심 사실을 모두 맞히고 중대한 모순이 없음.
- `PARTIAL`: 핵심 일부는 맞지만 수치·시점·관계·요구 항목 일부가 빠지거나 잘못됨.
- `FAIL`: 핵심 답을 하지 못하거나 반대로 답함.
- Legacy의 30~58건짜리 반복 목록은 답변 헤더와 질문 관련 원문 행을 그대로 싣고, 무관한 행만 `[무관한 목록 생략]`으로 표시했다. 전체 응답은 원본 JSONL이 기준이다.

> - 파인튜닝에 필요한 최소 500~1000쌍의 질의-답변 학습 데이터 준비에 2~3주 이상이 필요하며, 데이터 라벨링과 정제 공수가 크다.

> - 고객사가 2주 내 1차 개선 결과를 요청해 빠른 개선이 우선되어야 한다.
> - 파인튜닝은 추후 2차 유지보수 또는 계약 연장 시 별도 과제로 검토한다.

**Agentic 실제 답변 · PASS · `search_project_evidence`**

> **파인튜닝은 학습 데이터 준비에 최소 2주 이상이 필요하고, 데이터 라벨링 및 정제 공수가 크며, 현재 팀 가용 리소스로는 2주 내 1차 개선 목표를 달성하면서 병행하기 불가능하여 이번 유지보수에서 제외하기로 결정되었다.**
>
> 근거:
> - 최소 500~1000쌍의 고품질 학습 데이터가 필요하고 정제·라벨링 시간이 크다.
> - 데이터 준비와 학습 실험에 2~3주 이상 걸리며 팀은 RAG 개선과 서버 안정화를 병행 중이다.
> - 고객사가 2주 내 1차 개선 결과를 요청했다.
> - 2차 유지보수 또는 계약 연장 시 별도 과제로 검토하기로 했다.

**차이·원인**

### A4

**질문**
re-ranking을 정식 도입하기로 결정한 근거 수치는?

**골든 답변**
환불 영역 정확도 75%→88%(+13%p), 응답 지연 +0.6초.

**참고 문항**
6/23 회의록 (결정 1)

**Legacy 실제 답변 · PARTIAL · `semantic`**

> **환불 영역 정확도가 75%에서 88%로 13%p 향상된 것과 응답 시간 증가가 평균 0.6초로 허용 가능한 수준이라는 점이다.**
>
> - 예비 테스트에서 정확도가 13%p 상승했다.
> - 응답 시간은 평균 0.6초 증가했으나, **기존 평균 2.3초 대비 2.5초 수준**으로 허용 가능했다.

**Agentic 실제 답변 · PASS · `search_project_evidence`**

> re-ranking을 정식 도입하기로 결정한 근거 수치는 예비 테스트에서 환불 영역 정확도가 75%에서 88%로 13%p 향상되었고, 응답 지연 증가가 평균 0.6초로 허용 가능한 수준이었기 때문입니다. **즉, 13%p 정확도 향상과 0.6초 응답 지연 증가가 도입 근거 수치입니다.**

**차이·원인**
Legacy는 질문이 요구한 두 수치는 맞혔지만, 부연에서 `2.3초 + 0.6초 = 2.5초`처럼 산술적으로 맞지 않는 수치를 섞었다. 서로 다른 시점의 응답 시간 기록을 한 문장에 결합한 생성 오류다. Agentic은 요구된 두 근거 수치만 답해 모순을 피했다.

## 최종 원인 분류와 변경 우선순위

### 1. 가장 먼저 바꿀 것: 라우팅

### 2. 다음으로 바꿀 것: 검색 커버리지와 시점 정렬
