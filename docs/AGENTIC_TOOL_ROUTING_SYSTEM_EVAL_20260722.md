# Agentic Tool Routing 시스템 평가 보고서

- 평가일: 2026-07-22 (KST)
- 비교 대상: 기존 `legacy` 라우터 vs 단일 LLM 오케스트레이터 + 검색 도구(`agentic`)
- 메인 평가: Modu 30문항 + CS Bot 30문항, 총 60문항
- 원칙: 이 보고서는 **평가 당시 결과**와 **평가 후 별도 smoke에서 발견한 문제**를 분리한다. 후속 수정안은 제안일 뿐, 이 보고서 작성 과정에서는 코드를 변경하지 않았다.

## 1. 결론

이번 실험은 두 가지를 동시에 보여줬다.

1. **특정 사실을 묻는 semantic Q&A 경로는 유지할 가치가 있다.** 60문항 수기 재검토 기준 exact PASS는 54/60(90.0%)이었고, 기존 라우터에서 발생한 SQL 행 덤프형 답변을 제거했다. 평균 응답 길이는 1,353.8자에서 237.4자로 82.5% 줄었고, 평균 지연도 6.97초에서 6.27초로 10.0% 줄었다.
2. **현재 상태를 시스템 전체에 적용하거나 병합하면 안 된다.** 메인 60문항의 `expected_route`가 전부 `semantic`이라 구조화 list/count와 overview 선택 능력을 평가하지 못했다. 별도 실제 API smoke에서는 `category` 누락으로 액션 개수를 전체 memory 개수로 답하는 P0와, `completed_at` 데이터가 실제 완료 상태를 반영하지 않아 완료/미완료 답변이 틀리는 문제가 확인됐다.

따라서 판정은 **“실험 브랜치 유지, 운영 병합 보류”**다. semantic 도구 경로의 성과는 보존하되, route-balanced 평가셋과 구조화 데이터 정합성 검증을 먼저 통과해야 한다.

## 3. 평가 당시 구현 스냅샷

비첨부 프로젝트 Q&A의 실험 경로는 다음 구조였다.

1. 상위 LLM이 첫 턴에 반드시 하나 이상의 검색 도구를 호출한다.
2. 사용할 수 있는 도구는 세 가지다.
   - `search_project_evidence`: 특정 사실, 담당자, 수치, 이유, 변화 이력 검색
   - `query_structured_memory`: 조건이 명확한 목록/개수 조회
   - `get_project_overview`: 프로젝트 전반 브리핑
3. 도구는 근거만 반환하고, 최종 답변은 같은 오케스트레이터 LLM이 작성한다.
4. 최대 도구 라운드는 2회다.
5. 도구 호출 자체가 실패하면 기존 hybrid semantic RAG로 폴백한다.
6. 외부 API 호환을 위해 응답의 최상위 `route`는 `semantic`으로 유지하고, 실제 선택은 `debug.tools_used`, `debug.tool_calls`, `debug.tool_results`에 기록한다.
7. 첨부 파일이 있는 질문은 이 실험 경로가 아니라 기존 첨부 semantic 경로를 유지했다.

또한 agentic의 semantic 검색은 첫 LLM 호출에서 검색어 변형까지 만들고, retrieval 내부의 별도 multi-query LLM 호출을 생략했다. 이 차이가 semantic 구간 지연 감소에 직접 영향을 줬다.

## 4. 메인 60문항 결과

### 4.1 정답 정합성

자동 judge 원본은 PASS 50, PARTIAL 8, FAIL 2였다. 하지만 자동 judge가 CS C8의 문서상 2026-06-24를 근거 없이 2024년이어야 한다고 판정하는 등 판정 오류가 있어, 자동 결과를 최종 정답률로 사용하지 않았다.

질문·참고답변·검색 근거를 함께 대조한 수기 재검토 결과는 다음과 같다.

| 코퍼스 | PASS | PARTIAL | FAIL | exact PASS율 |
|---|---:|---:|---:|---:|
| Modu | 28 | 0 | 2 | 93.3% |
| CS Bot | 26 | 3 | 1 | 86.7% |
| 합계 | 54 | 3 | 3 | **90.0%** |

PARTIAL을 임의로 0.5점 처리하지 않았다. 따라서 이 보고서의 “90.0%”는 **완전 정답만 센 exact PASS율**이다.

확인된 핵심 오차는 다음 여섯 건이다.

| 코퍼스/QID | 판정 | 관찰된 문제 | 원인 층위 |
|---|---|---|---|
| Modu B4 | FAIL | 게시판형 MVP 유지 + 실시간 채팅 1.1 편입을 “기존 결정 번복”으로 표현 | 관계/시점 합성 오류 |
| Modu C1 | FAIL | 앱 SDK 연동과 백엔드 OAuth 지원의 담당 경계를 섞어 Apple 담당자를 잘못 답함 | 대상 경계 혼동 |
| CS B6 | FAIL | 정답 1.9초→2.5초 대신 2.3초→2.9초를 답함 | 검색 recall 실패 후 잘못된 수치 합성 |
| CS B7 | PARTIAL | 미정→구체화 흐름은 잡았으나 “일관되게 결정”처럼 초기 상태를 과장 | 관계 표현 정밀도 부족 |
| CS B10 | PARTIAL | 전체 fallback 18%는 답했지만 반품/교환 31%를 누락 | 복수 요구사항 중 일부 누락 |
| CS C5 | PARTIAL | 미확정은 맞지만 예정일을 이미 시작/확정된 사실처럼 과장 | 예정과 완료 상태 혼동 |

CS B6은 라우팅 문제가 아니었다. `search_project_evidence`는 선택됐지만, 선택된 청크에 6/23의 정확한 “1차 개선 후 1.9초” 근거가 빠지고 이전 2.3초와 re-ranking 증가분 0.6초가 결합돼 2.9초가 만들어졌다. 즉 이 건은 **도구 선택 성공, retrieval coverage 실패**다.

주의할 점은 결과 디렉터리에 legacy 답변용 동일 judge 파일이 없다는 것이다. 따라서 이 시스템 보고서만으로 “정답률이 legacy 대비 몇 %p 상승했다”고 주장하지 않는다. 위 90.0%는 agentic의 절대 평가이며, legacy 개별 정오 대조는 코퍼스별 질문 보고서에서 별도로 다뤄야 한다.

### 4.2 Route와 tool 선택

골든의 기대 라우트와 실제 선택은 다음과 같다.

| 항목 | Agentic | Legacy |
|---|---:|---:|
| expected_route=`semantic` | 60/60 | 60/60 |
| semantic/evidence 경로 선택 | 60/60 | 35/60 |
| `filter_lookup` 선택 | 0/60 | 20/60 |
| `overview` 선택 | 0/60 | 5/60 |

Agentic 내부 trace는 다음과 같다.

- 60문항 모두 `search_project_evidence` 사용
- 총 도구 호출 63회
  - 57문항: 1회 호출
  - 3문항: 한 라운드에서 대상별 2회 병렬 호출
- 60문항 모두 도구 라운드 1회에서 종료
- 도구 결과 63회 모두 `status=ok`
- agentic 실패 후 semantic 폴백: 0회

즉 메인 평가에서는 기대값과 도구 선택이 일치했다. 반대로 legacy는 25/60(41.7%)을 구조화 조회 또는 overview로 잘못 보냈다. 특히 “누가”, “몇 %”, “전체 정답률” 같은 표면 단어가 규칙/LLM 라우터를 자극해, 특정 사실 검색 질문을 전체 행 조회나 프로젝트 조망으로 오분류했다.

다만 이것은 **Agentic이 세 도구를 잘 구분했다는 증거가 아니다.** 60문항의 기대값이 하나의 route로만 구성됐기 때문에 `query_structured_memory`와 `get_project_overview`를 선택해야 하는 양성 사례가 메인 평가에 없었다. 실제 route 다양성은 아래 별도 smoke에서만 관찰했다.

참고로 `include_history`는 agentic trace에서 12문항이 true였지만 골든의 `expected_history_mode=true`는 5문항뿐이었고, 9문항에서 값이 달랐다. 일부 변화 질문은 골든 표기보다 history 검색이 합리적일 수 있어 단순 오답으로 세지는 않았지만, history mode도 별도 기준 정리가 필요하다.

## 5. 답변 길이와 폭주

여기서 “폭주”는 질문에 필요한 값 대신 광범위한 memory 행을 템플릿으로 나열해 답변이 비정상적으로 길어지는 현상으로 정의했다.

| 지표 | Agentic | Legacy | 변화 |
|---|---:|---:|---:|
| 평균 글자 수 | 237.4 | 1,353.8 | -82.5% |
| 중앙값 글자 수 | 240.5 | 330.0 | -27.1% |
| 최대 글자 수 | 491 | 7,360 | -93.3% |
| 평균 줄 수 | 4.78 | 19.37 | -75.3% |
| 최대 줄 수 | 9 | 105 | -91.4% |
| 1,000자 초과 | 0/60 | 16/60 | 16건 제거 |
| 3,000자 초과 | 0/60 | 10/60 | 10건 제거 |
| `조건에 맞는 기록` 구조화 템플릿 | 0/60 | 20/60 | 20건 제거 |

Legacy의 `filter_lookup` 20건은 모두 구조화 템플릿 답변이었다. 이 중 16건이 1,000자를 넘었고, filter_lookup 평균 길이는 3,441.9자였다. 짧은 4건도 좋은 답변이라기보다 광범위하거나 잘못된 필터로 “조건에 맞는 기록이 없다”고 반환한 사례가 포함된다.

따라서 길이 감소는 단순 문체 변화가 아니다. 기존에는 라우팅 오류가 곧 “전체 SQL 행 노출”로 증폭됐지만, agentic에서는 질문에 맞는 근거를 검색한 뒤 최종 LLM이 필요한 필드만 골라 답했다. 사용자에게 보였던 “제대로 집어서 말하지 않고 우다다 쏟아내는” 현상은 이 경로 차이로 설명된다.

아래 표는 동일 QID를 legacy가 어떤 route로 보냈는지에 따라 묶은 뒤, 같은 문항의 agentic 지연과 비교한 것이다.

해석은 다음과 같다.

- Legacy의 filter/overview 구간은 빨랐지만, 이 25문항의 기대 route는 전부 semantic이었다. 빠른 이유는 정확한 근거 검색과 최종 합성을 생략하고 템플릿 행 조회 또는 얕은 overview 답변으로 끝냈기 때문이다.
- 실제 semantic 대 semantic 비교에서는 agentic이 평균 40.1% 빨랐다. Agentic 첫 호출이 도구 선택과 검색어 생성을 함께 수행해 legacy semantic 경로의 별도 라우터 + 내부 multi-query 생성 호출을 줄인 효과가 크다.
- 따라서 전체 평균 개선은 “모든 질문이 항상 빨라졌다”가 아니라, semantic 경로를 단순화한 이득이 오분류된 빠른 legacy 경로의 비용 차이를 상쇄한 결과다.

## 7. 별도 실제 API route smoke

메인 60문항에 없는 list/count/overview 질문을 실제 API에 넣어 관찰했다. 이 결과는 **메인 60문항 점수에 포함되지 않는다.**

### 7.1 도구 선택 자체

| 질문 | 선택 도구 | 관찰 |
|---|---|---|
| `박현우가 담당한 미완료 액션 목록 보여줘` | `query_structured_memory(list)` | 총 15건 중 8건 반환, `truncated=true`. 출력 상한은 동작했지만 상태/분류 정합성 문제로 내용의 정확성은 보장되지 않음 |
| `완료된 액션 몇 개야?` | `query_structured_memory(count)` | 0건으로 답함. 실제 기록 부재가 아니라 `completed_at` 미적재 문제였음 |
| `프로젝트 전체 현황을 요약해줘` | `get_project_overview` | 의도한 overview 도구 선택 |

이 세 건은 LLM이 semantic 도구에 고착된 것은 아니며, 질문 형태에 따라 세 도구를 선택할 수 있음을 보여준다. 그러나 표본이 각 1건 수준이고 아래 P0가 발생했으므로 route 정확도라고 부를 수 없다.

### 7.2 평가 후 발견한 P0: `category` 인자 누락

후속 실제 API smoke에서 다음이 확인됐다.

| 질문 | 실제 tool call/결과 | DB 기준 | 영향 |
|---|---|---:|---|
| `현재 미완료 액션은 총 몇 개인가?` | `query_structured_memory(count, completed=false)`에서 `category` 누락, 전체 memory **103개**를 액션 수로 답함 | action **55개** | 전체 카테고리 개수를 액션 개수로 오답 |
| `김태호가 담당한 결정 목록` | 간헐적으로 `category` 누락, 담당자 조건의 전체 카테고리 **45개** 조회 | decision **24개** | 결정 외 action/issue/risk가 목록에 섞임 |

원인은 프롬프트가 “액션이면 action, 결정이면 decision을 넣으라”고 설명하더라도, 평가 당시 실제 tool boundary가 누락을 확실히 거부하지 못했다는 데 있다. LLM 지침은 데이터 계약이 아니다. `category=None`이 `mysql_search.search()`로 내려가면 SQL의 category 조건 자체가 빠지고 전체 카테고리를 조회한다.

이 문제는 P0다. 짧고 자연스러운 최종 답변이라도 내부 집계 대상이 틀리면 사용자는 오류를 알아차리기 어렵다. 또한 메인 60문항은 구조화 도구를 한 번도 선택하지 않았으므로 90.0% 정답률에 이 결함이 반영되지 않았다.

### 7.3 평가 후 발견한 데이터 정합성 문제: `completed_at`

평가용 snapshot에서는 액션 본문에 “기능 완료”, “설계 완료” 같은 표현이 있어도 관련 행의 `completed_at`이 NULL인 사례가 확인됐다. 그 결과 `completed=false`가 붙은 목록에 이미 완료됐다고 적힌 행이 “미완료”로 반환됐고, `completed=true` 개수는 0으로 나왔다.

현재 데이터 흐름의 문제는 다음과 같다.

1. 스키마와 ingestor는 `completed=true`를 받으면 `completed_at`을 저장할 수 있다.
2. 그러나 일반 회의록 추출 지침은 이미 완료된 액션의 `completed` 상태를 일관되게 채우도록 충분히 강제하지 않는다. repo commit에는 별도 보정이 있지만 회의록에는 같은 결정적 보정이 없다.
3. 이후 회의에서 과거 작업의 완료를 보고해도 이전 open action을 찾아 완료 처리하는 일반 action lifecycle reconciliation이 없다.
4. 조회에서는 `completed_at IS NULL`을 곧바로 “미완료”로 해석한다. 실제로는 `NULL`이 “미완료”와 “상태 미추출/미확인”을 함께 뜻한다.
5. overview의 열린 액션/완료 액션 집계도 같은 필드를 사용하므로, 이 문제는 count/list뿐 아니라 프로젝트 전체 현황까지 오염시킨다.

따라서 현재 DB snapshot에서 완료/미완료 관련 답변은 route가 맞아도 신뢰할 수 없다. 반면 메인 semantic 질문은 원문 청크를 함께 검색했기 때문에 이 필드 결함의 영향을 일부 피했다. 이것 역시 메인 60문항 성과를 구조화 상태 질의 성능으로 일반화하면 안 되는 이유다.

## 8. 왜 이런 결과가 나왔는가

### 좋아진 이유

- 기존 정규식은 `누가`, `몇 개`, `전체` 같은 단어 하나를 route 신호로 사용해 질문의 실제 대상과 목적을 놓쳤다. Agentic은 전체 질문 의미를 보고 specific evidence 검색을 선택했다.
- 도구는 근거만 반환하고 최종 LLM이 답을 작성하므로, SQL 결과 전체를 사용자에게 그대로 노출하지 않았다.
- 검색어 생성과 도구 선택을 한 LLM 턴에 합쳐 semantic 경로의 중복 호출을 줄였다.
- 두 대상이 있는 일부 질문은 대상별 검색을 한 라운드에서 병렬 호출해 한쪽 누락 가능성을 줄였다.

### 아직 틀린 이유

- LLM 라우팅은 정규식보다 유연하지만 tool argument의 필수 조건까지 자동으로 보장하지 않는다. `category`처럼 결과 집합을 바꾸는 인자는 런타임 계약이 필요하다.
- retrieval이 정확한 시점의 청크를 놓치면 최종 LLM은 인접한 수치와 변화량을 그럴듯하게 결합할 수 있다(CS B6).
- 같은 문맥에 역할이 다른 유사 작업이 함께 나오면 앱 SDK와 백엔드 OAuth처럼 대상 경계를 섞을 수 있다(Modu C1).
- “유지하면서 후속 버전에 편입” 같은 시간 관계는 단순 키워드 검색만으로는 번복/구체화를 정밀하게 구분하기 어렵다(Modu B4, CS B7).
- 복수 수치 질문에서 하나만 검색되면 최종 답변도 일부만 맞는다(CS B10).
- 예정/계획과 완료/확정을 표현하는 데이터 모델과 답변 규칙이 약해 미래 상태를 과장한다(CS C5, `completed_at` 문제).

## 9. 코드 변경 전 제안

이 항목은 구현 지시가 아니라 다음 실험의 우선순위다.

### P0 — 구조화 tool contract

1. `category`를 `decision/action/issue/risk/all` 중 하나의 필수 enum으로 만들고 누락을 runtime에서 거부한다.
2. 질문에 `액션/결정/이슈/리스크`가 명시됐는데 `all`이 오면 실행하지 말고 tool 재호출을 요구한다.
3. tool artifact에 요청 필터와 실제 적용 필터, 카테고리별 원본 count를 남겨 오집계를 관측 가능하게 한다.
4. “미완료 액션 수”, “담당자 결정 목록” 실제 API 회귀 테스트를 추가해 103→55, 45→24 같은 범위 누락을 잡는다.

### P0 — 완료 상태 데이터

1. `NULL=미완료` 가정을 제거하고 `open/completed/unknown`의 3상태로 취급한다.
2. 일반 회의록 extractor가 명시적 완료 표현을 `completed=true`로 기록하게 하고 근거 문장/상태 시점을 보존한다.
3. 후속 문서의 완료 보고를 기존 open action과 연결해 이전 행을 완료 처리하는 reconciliation을 둔다.
4. 현재 적재 코퍼스를 backfill/audit한 뒤 알려진 action 총수·완료·미완료·unknown 수를 fixture로 고정한다.
5. 상태가 unknown인 동안 count 답변은 단일 숫자로 단정하지 않고 “DB상 확인된 완료 N건, 상태 미확인 M건”처럼 한계를 노출한다.

### P1 — 평가셋과 retrieval

1. semantic 60개와 별도로 list/count/overview 양성·음성·복합 질문을 균형 있게 구성한다.
2. route 평가는 최상위 `route` 필드가 아니라 `debug.tools_used`와 실제 tool args를 기준으로 한다.
3. 각 질문마다 route 정확도뿐 아니라 filter argument 정확도, 집계 대상 행, 최종 답변을 각각 채점한다.
4. 시점별 수치 질문은 시점마다 독립 근거가 회수됐는지 검사하고, 한쪽이 없으면 추가 검색 또는 불확실성 응답으로 끝낸다.
5. 앱 SDK/백엔드 OAuth 같은 유사 작업 경계, 유지/구체화/연기/번복 같은 관계 분류를 adversarial fixture로 추가한다.

## 10. 다음 평가의 통과 기준

운영 병합 전 최소 통과 기준을 다음처럼 제안한다.

- 기존 semantic 60문항 exact PASS 90% 이상 유지
- route-balanced 신규 셋에서 tool 선택 정확도 95% 이상
- 질문에 명시된 category 인자 정확도 100%
- count 질문은 DB 검증값과 100% 일치
- 완료/미완료 질문은 unknown을 오답으로 단정하는 사례 0건
- 1,000자 초과 비요청 답변 0건
- 잘못된 전체 행 덤프 0건
- agentic fallback과 tool error를 별도 지표로 노출
- semantic p95가 이번 13.64초보다 악화되지 않음

## 11. 최종 판단

정규식 기반 legacy 라우팅이 사용자 질문을 “조건 단어”로만 보고 SQL 출력으로 직행한 것이 답변 폭주의 주원인이었다. 단일 LLM이 검색 절차를 도구로 선택하게 한 방향은 semantic 프로젝트 Q&A에서 더 정확하고 훨씬 간결했으며, semantic 구간 지연도 줄였다.

하지만 이번 60문항은 semantic만 평가했다. 별도 smoke에서 드러난 `category` 누락과 `completed_at` 정합성은 구조화 list/count/overview 기능을 현재 상태로 신뢰할 수 없게 만드는 차단 결함이다. 따라서 이 실험을 폐기할 이유는 없지만, 지금 병합할 근거도 없다. **보고서를 기준선으로 고정하고 P0 데이터/계약 문제를 먼저 해결한 뒤, route-balanced 평가를 다시 실행하는 것이 맞다.**
