PaiM FastAPI 명세서

Base URL (로컬): http://127.0.0.1:8000  ·  API Prefix: /api/v1 (GitHub App은 /github/app/*, prefix 없음)  ·  에러 형식: {"detail": "..."}  ·  갱신: 2026-07-22 (TASK-010 — 누락 20 operation·인증/권한 최신화)

인증·권한

인증 모드·갱신일 단일 출처: 인증 정책은 이 섹션이 정본이다 (문서 상단 auth-mode·updated 표식과 값 일치). 서버는 기본 PAIM_AUTH_MODE=jwt(fail-closed) — 보호 엔드포인트는 Authorization: Bearer <token> 필수, 없거나 무효면 401. PAIM_JWT_SECRET(≥32바이트, 비플레이스홀더) 미설정 시 기동 거부. PAIM_AUTH_MODE=dev는 로컬 전용(DEV_USER_ID 대체, 배포 금지). 공개 경로: GET /, GET /health, POST /api/v1/auth/signup, POST /api/v1/auth/login, GET /github/app/callback(state 필수).
역할 서열: viewer < member < admin < owner (프로젝트별 project_members). 조회=viewer, 상태 변경(업로드·삭제· 메모리 CRUD·제안 accept/reject·git·세션 생성/수정/삭제)=member, 프로젝트 삭제=owner. 멤버 API: 멤버 목록 조회=viewer, 멤버 초대=owner, 역할 변경=owner, 멤버 제거는 본인 탈퇴=member· 타인 제거=owner. dev 모드는 DEV_USER_ID가 설정돼 있으면 실제 역할로 검사하고, 미설정일 때만 검사를 생략(no-op)한다. 엔드포인트별 최소 역할은 각 카드와 Markdown 명세서의 역할 표를 따른다.

인증 API

POST /api/v1/auth/signup
회원가입 (공개). 성공 시 access token·사용자 정보 반환.
Request Body
{ "email": "a@b.c", "password": "at-least-8-chars", "name": "이름" }
201 Created
{ "access_token": "<JWT>", "token_type": "bearer", "user": { "id": 1, "email": "a@b.c", "name": "이름" } }
400 이메일 형식 / 비밀번호 길이
409 이미 가입된 이메일

로그인 (공개). signup과 동일한 응답 형태.
Request Body
{ "email": "a@b.c", "password": "..." }
200 OK — { access_token, token_type, user }
401 이메일 또는 비밀번호가 올바르지 않습니다 (사유 미노출)
GET /api/v1/auth/me
현재 토큰의 사용자 정보 (인증 필요).
200 OK
{ "id": 1, "email": "a@b.c", "name": "이름", "created_at": "..." }
401 토큰 없음/무효/만료

Git Repository

GitHub repository 연결 및 초기 sync 시작. public repo는 token 불필요. private repo는 GitHub App state 파라미터로 token 전달.
Request Body

  "provider": "github",
  "repository_url": "https://github.com/org/repo",
  "branch": "main",
  "state": null

providerstring✅github 고정 (MVP)
repository_urlstring✅GitHub repo URL
branchstring-미지정 시 default branch 사용

201 Created
{ "repo_id": 3, "status": "syncing" }
400 Bad Request
{ "detail": "Invalid repository URL" }

LLM이 문서 또는 repository에서 추출한 구조화된 프로젝트 기억.

파라미터타입설명

datestring날짜 필터 YYYY-MM-DD
200 OK
[

    "id": 1,
    "project_id": 1,
    "doc_id": 12,
    "repo_id": null,
    "category": "action",
    "content": "API 규약을 정리한다",
    "owner": "지훈",
    "date": "2026-06-30",
    "topic": "API 설계",
    "reason": null,
    "source": "planning.pdf",
    "created_by": "llm",
    "updated_by": null,
    "is_user_verified": false,
    "created_at": "2026-07-01T12:00:00",

      "kind": "document",
      "doc_id": 12,
      "repo_id": null,
      "type": "meeting",
      "path": "planning.pdf",
      "ref": null,

역할 서열 viewer < member < admin < owner. owner는 프로젝트 생성 시에만 부여(이관 불가). 할당 가능 role: viewer/member/admin.

GET /api/v1/projects/{project_id}/members
멤버 목록 조회 (최소 역할 viewer).
200 OK
[{ "user_id": 1, "email": "a@b.c", "name": "이름", "role": "owner", "created_at": "...", "last_seen_at": null }]
POST /api/v1/projects/{project_id}/members
멤버 추가 (최소 역할 owner). 대상은 이미 가입된 사용자.
Request Body
{ "email": "new@b.c", "role": "member" }
201 Created
{ "user_id": 5, "email": "new@b.c", "name": "새 멤버", "role": "member" }
400 허용되지 않는 role  ·  404 미가입  ·  409 이미 멤버
PATCH /api/v1/projects/{project_id}/members/{member_user_id}
멤버 역할 변경 (최소 역할 owner).
Request Body
{ "role": "admin" }
200 OK — { "user_id": 5, "role": "admin" }
400 role 오류/본인 변경  ·  403 owner 변경 불가  ·  404 비멤버
DELETE /api/v1/projects/{project_id}/members/{member_user_id}
멤버 제외. 본인 탈퇴 = member, 타인 제거 = owner (진입 최소 역할 member).
204 No Content

제안 (Suggestions)

GET /api/v1/projects/{project_id}/suggestions
제안 목록 (최소 역할 viewer). Query: status(pending)·kind(complete_action).
200 OK
[{ "id": 8, "project_id": 1, "memory_id": 10, "kind": "complete_action",
  "evidence": {...}, "rationale": "...", "confidence": "high",

POST /api/v1/projects/{project_id}/suggestions/{suggestion_id}/accept
제안 승인 (최소 역할 member). complete_action=완료 처리, supersede=결정 번복.
200 OK — 갱신된 suggestion (status: "accepted")

POST /api/v1/projects/{project_id}/suggestions/{suggestion_id}/reject
제안 거절 (최소 역할 member).
200 OK — 갱신된 suggestion (status: "rejected")

상태 폴링 흐름

문서 처리

POST /documents → { "status": "processing" }
↓ GET /documents/{id}/status 반복
→ processing : 계속 대기
→ indexed : 처리 완료
→ failed : last_error 메시지 표시

Repository Sync

POST /repositories 또는 POST /repositories/{id}/sync → { "status": "syncing" }
↓ GET /repositories/{id}/status 반복
→ syncing : 계속 대기
→ indexed : 완료 (sync_warning 확인 권장)
→ failed : last_error 메시지 표시
서버 재시작 시 30분 이상 stale 상태인 processing/syncing은 자동으로 failed로 전환됨 (BACKGROUND_TASK_STALE_MINUTES 환경 변수로 조정 가능).

주의 사항

항목설명
sync_warning 타입 JSON-encoded string 또는 null. non-null이면 JSON.parse() 후 source_type별 reason 표시 권장
인증 정책은 상단 인증·권한 섹션이 정본이다(여기서는 중복 서술하지 않음).
권한 역할 서열 viewer < member < admin < owner. 조회(멤버 목록 포함)=viewer, 상태 변경=member, 프로젝트 삭제·멤버 초대/역할 변경=owner, 멤버 제거는 본인 탈퇴=member·타인 제거=owner. 상세는 인증·권한 역할 요약 참조. dev 모드 권한 검사 생략은 DEV_USER_ID 미설정 시에 한함.
stale recovery 서버 재시작 시 BACKGROUND_TASK_STALE_MINUTES(기본 30) 초과 stale → failed 자동 전환
원자성 MySQL + ChromaDB + 파일 간 완전 원자성 미보장. 실패 시 status/last_error로 추적, 재sync으로 복구
