문서

POST /api/v1/projects/{project_id}/documents
문서 업로드. 서버가 파일을 저장하고 텍스트 추출 → 청킹 → Vector DB → LLM 추출 → DB 저장 처리.
Request — multipart/form-data
필드타입필수설명
fileFile✅.md / .txt / .pdf (최대 10 MB)
datestring-문서 날짜 YYYY-MM-DD
doc_type 변경 (2026-07-02): 프론트에서 전송하지 않습니다. 서버가 파일명 기반으로 자동 추론합니다.
회의/meeting/minutes 포함 → meeting  |  기획/plan/planning/roadmap/spec 포함 → planning  |  그 외 → document
201 Created
{ "doc_id": 12, "status": "processing" }
400 Bad Request
{ "detail": "지원하지 않는 파일 형식입니다. (.md / .txt / .pdf)" }
413 Content Too Large
{ "detail": "파일 크기는 10 MB를 초과할 수 없습니다." }
403 Forbidden
{ "detail": "project access denied" }
업로드 즉시 processing 반환 → 백그라운드 처리 완료 후 indexed 또는 failed로 갱신. GET /documents/{doc_id}/status로 폴링하여 완료 확인.

Git Repository

GitHub repository 연결 및 초기 sync 시작. public repo는 token 불필요. private repo는 GitHub App state 파라미터로 token 전달.
Request Body

  "provider": "github",
  "repository_url": "https://github.com/org/repo",
  "branch": "main",
  "state": null
}

상태 폴링 흐름

문서 처리

POST /documents → { "status": "processing" }
↓ GET /documents/{id}/status 반복
→ processing : 계속 대기
→ indexed : 처리 완료
→ failed : last_error 메시지 표시

Repository Sync

POST /repositories 또는 POST /repositories/{id}/sync → { "status": "syncing" }
↓ GET /repositories/{id}/status 반복
→ syncing : 계속 대기
→ indexed : 완료 (sync_warning 확인 권장)
→ failed : last_error 메시지 표시
서버 재시작 시 30분 이상 stale 상태인 processing/syncing은 자동으로 failed로 전환됨 (BACKGROUND_TASK_STALE_MINUTES 환경 변수로 조정 가능).

공통 에러 코드

상태 코드의미
400잘못된 요청 (빈 content, 잘못된 날짜 형식 등)
403프로젝트 접근 권한 없음
404리소스 없음
422Pydantic 유효성 검사 실패
503LLM / 외부 서비스 오류

주의 사항

항목설명
sync_warning 타입 JSON-encoded string 또는 null. non-null이면 JSON.parse() 후 source_type별 reason 표시 권장
인증 현재 DEV_USER_ID env 기반 fallback. 운영 도입 시 Authorization 헤더 토큰으로 교체 예정
권한 write 작업(upload, delete, memory CRUD, repo connect/sync/delete)은 member 이상 필요. DEV 모드에서는 권한 체크 no-op
stale recovery 서버 재시작 시 BACKGROUND_TASK_STALE_MINUTES(기본 30) 초과 stale → failed 자동 전환
원자성 MySQL + ChromaDB + 파일 간 완전 원자성 미보장. 실패 시 status/last_error로 추적, 재sync으로 복구
